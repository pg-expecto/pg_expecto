## Функция `check_and_forget()`

### 1. Назначение

Функция `check_and_forget()` является **основной процедурой форсированного (адаптивного) забывания** в системе цепи Маркова. Она предназначена для автоматического обнаружения изменений в поведении системы (дрейф распределения, ухудшение качества прогнозов, аномалии операционной скорости, внешние события) и, при необходимости, принудительного применения забывания с динамически рассчитанным коэффициентом `alpha`.

В отличие от планового забывания, которое выполняется с фиксированным интервалом и `alpha` (внутри `markov_chain_training`), `check_and_forget()`:
- работает по расписанию (рекомендованный интервал – 15 минут);
- анализирует несколько статистических метрик;
- требует подтверждения признаков на протяжении нескольких циклов (`confirmation_cycles`);
- суммирует вклад сработавших признаков в итоговый `alpha`;
- логирует причины и метрики в таблицу `forget_log`.

Если адаптивное забывание отключено (`markov_config.adaptive_forgetting_enabled = false`), функция немедленно возвращает сообщение об этом без каких-либо действий.

### 2. Сигнатура

```sql
CREATE OR REPLACE FUNCTION check_and_forget()
RETURNS TEXT
LANGUAGE plpgsql
```

**Входные параметры:** отсутствуют (все настройки читаются из таблицы `markov_config`).

**Возвращаемое значение:** `TEXT` – диагностическое сообщение:
- `'Adaptive forgetting is disabled by markov_config.adaptive_forgetting_enabled = false.'` – если отключено;
- `'Forgetting applied with alpha=XXX, reasons: YYY'` – если забывание было выполнено;
- `'No forgetting needed.'` – если условия не достигнуты.

### 3. Логика работы (по шагам)

#### 3.1. Чтение конфигурации
Из таблицы `markov_config` извлекаются следующие параметры:
- `kl_threshold` – порог KL-дивергенции (по умолчанию 0.4);
- `chi2_threshold` – порог χ² (по умолчанию 220.0);
- `os_dev_threshold` – порог отклонения операционной скорости (по умолчанию 0.3);
- `brier_threshold` – порог Brier Score (по умолчанию 0.25);
- `forget_alpha_max` – максимально допустимый `alpha` при суммировании (по умолчанию 0.5);
- `confirmation_cycles` – количество последовательных проверок, необходимых для срабатывания (по умолчанию 2);
- `adaptive_forgetting_enabled` – флаг включения механизма (по умолчанию `true`).

#### 3.2. Проверка включения адаптивного забывания
Если `adaptive_forgetting_enabled = false`, функция возвращает текст отключения и завершается.

#### 3.3. Очистка устаревших записей `check_state`
Удаляются записи из таблицы `check_state` старше 7 дней (для предотвращения неограниченного роста).

#### 3.4. Вычисление метрик

| Метрика | Функция | Окно | Назначение |
|---------|---------|------|-------------|
| **KL-дивергенция** (`kl_val`) | `calculate_kl_divergence(60, NULL, NULL)` | последние 60 минут | Сравнивает распределение состояний за последний час с эталонным (почасовым, подневным). Высокое значение (`> kl_threshold`) указывает на дрейф. |
| **χ²-критерий** (`chi2_val`) | `calculate_chi_squared(60, NULL, NULL)` | последние 60 минут | Аналогично KL, но на основе хи-квадрат. Высокое значение (`> chi2_threshold`) говорит о значимом расхождении. |
| **Отклонение операционной скорости** (`os_dev`) | `get_os_deviation()` | SMA20 за 20 минут vs историческое среднее за час | Выявляет аномалии производительности (например, падение скорости). |
| **Brier Score** (`brier_val`) | `AVG((predicted_risk - actual_risk)^2)` из `forecast_log` | последние 2 часа | Оценивает калибровку вероятностных прогнозов. Высокое значение (`> brier_threshold`) говорит об ухудшении качества модели. |
| **Флаг внешних событий** (`infra_flag`) | проверка `infrastructure_events` | последний час | Если есть необработанные события (деплой, сбой, изменения), модель должна быстрее забыть старую историю. |
| **Внутридневной дрейф** (`diurnal_flag`) | `calculate_kl_divergence(60, current_hour, NULL) > 0.2` | последние 60 минут | Сравнивает распределение текущего часа с эталоном того же часа. Высокая дивергенция может требовать небольшого ускорения забывания. |

#### 3.5. Сохранение флагов в `check_state`
Для каждого цикла проверки вставляется запись в таблицу `check_state` с полями:
- `check_time` – время проверки;
- `kl_flag`, `chi2_flag`, `os_flag`, `brier_flag`, `infra_flag`, `diurnal_flag` – булевы значения превышения соответствующих порогов.

#### 3.6. Подсчёт последовательных срабатываний
Выбираются последние `confirmation_cycles` записей из `check_state` (упорядоченных по убыванию `check_time`). Подсчитывается количество записей, в которых хотя бы один из флагов равен `true`. Если это количество равно `confirmation_cycles` (т.е. на протяжении всех последних N проверок система была в «тревожном» состоянии), то условие считается выполненным.

#### 3.7. Расчёт `alpha_eff` при выполнении условия
Если условие подтверждения выполнено, инициализируется `alpha_eff = 0` и пустой массив `reason_list`. Для каждого сработавшего признака добавляется вклад:

| Признак | Добавка к `alpha_eff` | Дополнительные действия |
|---------|----------------------|--------------------------|
| `kl_flag` | +0.1 | добавляет `'KL'` в `reason_list`, сохраняет `kl_val` в `details` |
| `chi2_flag` | +0.1 | добавляет `'Chi2'`, сохраняет `chi2_val` |
| `os_flag` | +0.1 | добавляет `'OS'`, сохраняет `os_dev` |
| `brier_flag` | +0.1 | добавляет `'Brier'`, сохраняет `brier_val` |
| `infra_flag` | `GREATEST(alpha_eff, 0.3)` (устанавливает минимум 0.3) | добавляет `'Infra'`, помечает все события за последний час как `processed = true` |
| `diurnal_flag` | +0.02 | добавляет `'Diurnal'` |

После суммирования:
- `alpha_eff = LEAST(alpha_eff, forget_alpha_max)` – ограничение сверху.
- Если `alpha_eff > 0.0`, вызывается `apply_forgetting(alpha_eff)` для реального применения забывания.

#### 3.8. Логирование в `forget_log`
После успешного применения забывания вставляется запись в таблицу `forget_log` с полями:
- `alpha` = `alpha_eff`;
- `triggered_by` = массив причин;
- `kl_div`, `chi2_val`, `brier_score`, `os_deviation` – численные значения метрик;
- `details` – текстовая строка, содержащая значения метрик для отладки.

#### 3.9. Возврат сообщения
- Если забывание применено: `'Forgetting applied with alpha=... , reasons: KL,Chi2,...'`
- Если условий недостаточно: `'No forgetting needed.'`

### 4. Вспомогательные функции

| Функция | Описание | Примечания |
|---------|----------|-------------|
| `calculate_kl_divergence(recent_minutes, baseline_hour, baseline_dow)` | Возвращает KL-дивергенцию между распределением состояний за последние `recent_minutes` минут и эталоном. | Использует таблицу `state_baseline`. При нулевых вероятностях применяет аддитивное сглаживание `1e-6`. |
| `calculate_chi_squared(recent_minutes, baseline_hour, baseline_dow)` | Возвращает χ²-статистику. | Ожидаемые частоты вычисляются как `probability * total_obs`. |
| `get_os_deviation()` | Возвращает относительное отклонение SMA20 операционной скорости от исторического среднего. | Использует `cluster_stat_median` и `operational_speed_stats`. |
| `apply_forgetting(alpha)` | Применяет забывание с заданным коэффициентом. | Обновляет `markov_frequencies`, пересчитывает `markov_probabilities` и `markov_absorbing`, обновляет `last_forget_time`. |

### 5. Используемые таблицы

| Таблица | Роль в `check_and_forget()` |
|---------|-----------------------------|
| `markov_config` | Хранит все пороги, `confirmation_cycles`, `forget_alpha_max`, флаг включения. |
| `check_state` | Сохраняет флаги каждой проверки для механизма подтверждения. |
| `transition_log` | Используется в `calculate_kl_divergence` и `calculate_chi_squared` для подсчёта наблюдаемых частот. |
| `state_baseline` | Эталонные распределения состояний по часам и дням недели. |
| `forecast_log` | Используется для вычисления Brier Score за последние 2 часа. |
| `infrastructure_events` | Таблица внешних событий; после обработки флаг `processed` устанавливается в `true`. |
| `operational_speed_stats` | Хранит среднюю скорость и std для каждого часа (для `get_os_deviation`). |
| `cluster_stat_median` | Исходные данные по операционной скорости (для SMA20). |
| `forget_log` | Журнал форсированных забываний; функция вставляет записи. |

### 6. Параметры конфигурации (из `markov_config`)

| Параметр | Тип | По умолчанию | Назначение |
|----------|-----|--------------|-------------|
| `kl_threshold` | REAL | 0.4 | Порог KL-дивергенции. Превышение → признак `kl_flag`. |
| `chi2_threshold` | REAL | 220.0 | Порог χ² (для 188 степеней свободы, p=0.01). Превышение → `chi2_flag`. |
| `os_dev_threshold` | REAL | 0.3 | Порог относительного отклонения скорости (30%). Превышение → `os_flag`. |
| `brier_threshold` | REAL | 0.25 | Порог Brier Score. Превышение → `brier_flag`. |
| `forget_alpha_max` | REAL | 0.5 | Верхняя граница для `alpha_eff`. |
| `confirmation_cycles` | SMALLINT | 2 | Сколько последовательных проверок должно быть тревожными для срабатывания. |
| `adaptive_forgetting_enabled` | BOOLEAN | true | Глобальный выключатель механизма. |

### 7. Возвращаемые значения и примеры

```sql
-- Если забывание отключено
SELECT check_and_forget();
-- Результат: 'Adaptive forgetting is disabled by markov_config.adaptive_forgetting_enabled = false.'

-- Если условия не выполнены
SELECT check_and_forget();
-- Результат: 'No forgetting needed.'

-- Если забывание применено (например, сработали KL и OS)
SELECT check_and_forget();
-- Результат: 'Forgetting applied with alpha=0.200, reasons: KL,OS'
```

### 8. Рекомендации по эксплуатации

#### 8.1. Настройка расписания
Рекомендуется запускать `check_and_forget()` каждые 15 минут с помощью pg_cron или внешнего планировщика:
```sql
-- Добавить задание (пример для pg_cron)
SELECT cron.schedule('check-and-forget', '*/15 * * * *', 'SELECT check_and_forget()');
```

#### 8.2. Калибровка порогов (первые 1–2 недели)
- Временно закомментировать внутри `check_and_forget()` вызов `PERFORM apply_forgetting(alpha_eff)`, но оставить все вычисления и запись в `forget_log`.
- Накопить журнал `forget_log` (без фактического забывания).
- Проанализировать распределение метрик, установить пороги так, чтобы ложные срабатывания происходили не чаще 1–2 раз в день.
- Для χ² использовать 99-й процентиль распределения с соответствующим числом степеней свободы (188 для 189 состояний).
- Для Brier Score начальное значение 0.25; после калибровки можно скорректировать до 0.2–0.3.

#### 8.3. Борьба с ложными срабатываниями
Механизм `confirmation_cycles` (по умолчанию 2) уже снижает чувствительность. При частых ложных срабатываниях можно увеличить до 3 (45 минут подтверждения). Также можно увеличить пороги или добавить дополнительную фильтрацию.

#### 8.4. Внутридневной дрейф
Признак `diurnal_flag` срабатывает при `calculate_kl_divergence(60, current_hour, NULL) > 0.2`. Это мягкий признак, добавляющий только +0.02 к `alpha_eff`. Если внутридневные изменения очень сильны, можно рассмотреть переход к множественным матрицам (разные слоты времени) или повысить частоту планового забывания в часы пик.

#### 8.5. Интеграция с внешними событиями
Таблица `infrastructure_events` должна пополняться через триггеры CI/CD или вручную. После обработки событий флаг `processed` устанавливается в `true`, чтобы не вызывать забывание повторно.

#### 8.6. Мониторинг
- Настроить алерт, если забывания происходят чаще 6 раз в сутки (можно менять порог). Это может указывать на избыточную чувствительность или реальную нестабильность.
- Еженедельно запускать `evaluate_training_sufficiency()` для проверки стабильности модели в целом.
- Периодически анализировать `forget_log` для выявления наиболее частых триггеров.

### 9. Пример полного цикла работы

1. В 14:00 запускается `check_and_forget()`.
2. Вычисляет KL=0.35 (порог 0.4) → `kl_flag = false`, χ²=210 → `chi2_flag = false`, OS deviation=0.32 → `os_flag = true`, Brier=0.24 → `brier_flag = false`, инфра-событий нет, диурнальный KL=0.15.
3. В `check_state` вставляется запись с `os_flag = true`, остальные `false`.
4. Смотрит последние 2 записи: предыдущая также имела `os_flag = true` → `consecutive_count = 2` (равно `confirmation_cycles`).
5. `alpha_eff = 0.1` (только OS), ограничение 0.5 не меняет.
6. Вызывает `apply_forgetting(0.1)`.
7. Вставляет запись в `forget_log` с `triggered_by = {'OS'}`, `os_deviation = 0.32`.
8. Возвращает `'Forgetting applied with alpha=0.100, reasons: OS'`.

### 10. Возможные ошибки и обработка

- **Отсутствие данных** – если за окно нет переходов в `transition_log`, KL и χ² возвращают `NULL`, соответствующие флаги `false`.
- **Отсутствие `state_baseline`** – функции `calculate_kl_divergence` и `calculate_chi_squared` вернут `NULL`, флаги не сработают.
- **Отсутствие `forecast_log`** – Brier Score будет 0.0 (через `COALESCE`), что не превысит порог (если порог >0).
- **Проблемы с `cluster_stat_median`** – `get_os_deviation()` вернёт 0.0, флаг `os_flag` не сработает.
- **Переполнение `alpha_eff`** – ограничивается `forget_alpha_max`, так что даже при срабатывании всех признаков `alpha_eff` не превысит заданный максимум.

### 11. Отладка

Для проверки работы без реального забывания можно временно закомментировать строку `PERFORM apply_forgetting(alpha_eff);` внутри функции. Тогда `check_and_forget()` будет только вычислять метрики, сохранять флаги и записывать в `forget_log` (если изменить код, чтобы вставка в `forget_log` происходила даже при `alpha_eff > 0`). Это безопасный способ калибровки.

Также полезно выполнять вручную:
```sql
-- Посмотреть последние записи check_state
SELECT * FROM check_state ORDER BY check_time DESC LIMIT 10;

-- Посмотреть журнал забываний
SELECT * FROM forget_log ORDER BY ts DESC LIMIT 10;
```

### 12. Заключение

Функция `check_and_forget()` реализует интеллектуальный механизм адаптивного забывания, который позволяет модели цепи Маркова автоматически подстраиваться под изменяющиеся условия, не дожидаясь планового интервала. Благодаря использованию нескольких статистических тестов, подтверждению признаков и логированию, она даёт как надёжную адаптацию, так и полную наблюдаемость процесса. При правильной калибровке порогов и настройке расписания функция становится ключевым элементом поддержания точности прогнозов в динамичных средах.
