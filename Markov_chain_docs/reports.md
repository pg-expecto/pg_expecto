# Отчёты и мониторинг цепи Маркова

В этом документе описаны аналитические и диагностические функции для оценки качества прогнозов, анализа аварийных переходов и мониторинга состояния модели.

---

## 1. mchain_reliability_report

**Назначение**  
Формирует расширенный текстовый отчёт о достоверности прогнозов модели на основе трёх ключевых метрик: объём накопленных данных, стабильность вероятностей переходов и покрытие частых состояний. Итоговый рейтинг от 0 (полностью недостоверен) до 5 (максимально достоверен).

**Сигнатура**  
```sql
mchain_reliability_report()
RETURNS TEXT
```

**Входные параметры**  
Нет.

**Возвращаемое значение**  
Многострочный текст (TEXT) с отчётом, содержащим:

- Общий рейтинг достоверности (0–5) и его интерпретацию.
- Общее число переходов (`total_transitions`) и порог `min_transitions_for_forgetting` из конфигурации.
- Максимальное изменение вероятностей за последние две недели (`max_prob_change`) – показатель стабильности.
- Покрытие частых состояний (`coverage_pct`) – доля состояний с частотой >1%, для которых накоплено не менее 50 переходов.
- Рекомендации по улучшению модели или её использованию.

**Логика работы**

1. **Объём данных** – оценивает `total_transitions`:
   - `< 100` → рейтинг 0
   - `100–499` → рейтинг 1
   - `500–4999` → рейтинг 2
   - `≥ 5000` → базовая часть рейтинга = 3 (далее добавляются бонусы)

2. **Стабильность** – сравнивает вероятности переходов за две последовательные недели (`now()-14d … now()-7d` и `now()-7d … now()`):
   - Изменение < 2% → бонус +2
   - Изменение < 5% → бонус +1
   - Иначе → бонус 0

3. **Покрытие частых состояний** – для состояний, встречающихся в более чем 1% всех переходов, проверяет, имеют ли они минимум 50 наблюдений:
   - Покрытие ≥ 90% → бонус +1
   - Иначе → бонус 0

4. Итоговый рейтинг = базовая оценка + бонусы (максимум 5).

5. Формируется читаемый отчёт с пояснениями и рекомендациями (например, «продолжить обучение», «прогнозы можно использовать с осторожностью»).

**Пример вызова**
```sql
SELECT mchain_reliability_report();
```

**Пример вывода (фрагмент)**
```
ОТЧЁТ О ДОСТОВЕРНОСТИ ПРОГНОЗОВ ЦЕПИ МАРКОВА
--------------------------------------------------------------------

1. ОБЩИЙ РЕЙТИНГ ДОСТОВЕРНОСТИ (0-5): 4
   Интерпретация: хорошая достоверность. Прогнозам можно доверять в большинстве ситуаций.

--------------------------------------------------------------------

2. ДЕТАЛИЗАЦИЯ ПО МЕТРИКАМ

   Общее число переходов (total_transitions): 12500
   Рекомендуемое минимальное значение: 5000 (min_transitions_for_forgetting)
   Статус: ДОСТАТОЧНО – модель имеет необходимый объём данных.

   Максимальное изменение вероятностей за две недели (max_prob_change): 0.032
   Порог стабильности: 0.05
   Статус: СТАБИЛЬНО – вероятности существенно не меняются.

   Покрытие частых состояний (coverage_pct): 95%
   Рекомендуемое покрытие: не менее 90%
   Статус: ХОРОШЕЕ – большинство частых состояний имеют достаточную статистику.

--------------------------------------------------------------------

3. РЕКОМЕНДАЦИИ
   - Прогнозы можно использовать с осторожностью, особенно при высоком риске.
   - Для достижения максимальной достоверности следует улучшить стабильность вероятностей и покрытие частых состояний.
```

**Примечания**  
- Функция **не зависит от периода** – оценивает всю доступную историю.
- Для расчёта стабильности и покрытия требуется не менее 5000 переходов; при меньшем объёме эти разделы отчёта пропускаются.

## 2. mchain_incident_transitions_report

**Назначение**  
Анализирует переходы в **аварийные состояния** (определяются как `correlation < 0 AND os_trend = -1`) за заданный временной интервал. Отчёт объединяет:

- Сырую статистику из журнала `transition_log` (фактическая частота аварий).
- Взвешенные частоты из таблицы `markov_frequencies` (текущая модель, учитывающая забывание).

Позволяет сравнить, насколько модель с забыванием соответствует реальной истории.

**Сигнатура**  
```sql
mchain_incident_transitions_report(
    p_start TIMESTAMPTZ DEFAULT NULL,
    p_end   TIMESTAMPTZ DEFAULT NULL
)
RETURNS TEXT
```

**Входные параметры**  
| Параметр | Тип | По умолчанию | Описание |
|----------|-----|--------------|-----------|
| `p_start` | TIMESTAMPTZ | `now() - interval '7 days'` | Начало анализируемого периода |
| `p_end`   | TIMESTAMPTZ | `now()` | Конец периода |

**Возвращаемое значение**  
Текстовый отчёт, содержащий:

- Период анализа.
- Общее количество переходов и число переходов в аварию (с процентной долей).
- Количество различных аварийных состояний, в которые были переходы.
- Суммарный вес переходов в модели `markov_frequencies` и долю аварийного веса.
- Сравнение доли аварий в сырых данных и в модели (разница, интерпретация).
- Динамику по часам (если период ≤ 2 дней) или по дням (если период длиннее).
- Рекомендации в зависимости от доли аварий и расхождения с моделью.

**Логика работы**

1. Определяет список аварийных состояний через `state_descriptions`.
2. Выполняет агрегацию `transition_log` за указанный период: общее число переходов, число аварийных, уникальные аварийные состояния.
3. Вычисляет суммарный вес из `markov_frequencies` и вес, приходящийся на аварийные переходы.
4. Формирует динамические срезы (почасовые или подневные) в зависимости от длины периода.
5. Сравнивает проценты и выдаёт диагностику:
   - `diff_pct > 5` → модель занижает риск (старые аварии «забыты»)
   - `diff_pct < -5` → модель завышает риск (свежие аварии имеют больший вес)
6. Выдаёт рекомендации по настройке `alpha`, `half_life` или периода забывания при значительном расхождении.

**Пример вызова**
```sql
-- Отчёт за последние 7 дней (по умолчанию)
SELECT mchain_incident_transitions_report();

-- За конкретный месяц
SELECT mchain_incident_transitions_report('2025-05-01', '2025-06-01');
```

**Пример вывода (фрагмент)**
```
ОТЧЁТ О ПЕРЕХОДАХ В АВАРИЙНЫЕ СОСТОЯНИЯ
--------------------------------------------------------------------

Период: 2025-05-25 00:00:00 – 2025-06-01 00:00:00

--------------------------------------------------------------------
1. СЫРАЯ СТАТИСТИКА (transition_log)
   Всего переходов: 10080
   Переходов в аварию: 87 (0.86% от общего числа)
   Различных аварийных состояний: 12

--------------------------------------------------------------------
2. ТЕКУЩАЯ МОДЕЛЬ (markov_frequencies с учётом забывания)
   Суммарный вес всех переходов: 8450.23
   Вес аварийных переходов: 92.15 (1.09% от общего веса)

--------------------------------------------------------------------
3. СРАВНЕНИЕ И АНАЛИЗ
   Доля аварийных переходов в сырой истории: 0.86%
   Доля аварийных переходов в модели (с забыванием): 1.09%
   Разница: -0.23% (модель оценивает риск ВЫШЕ исторического)
   → Оценки близки – модель хорошо отражает историческую частоту аварий.

--------------------------------------------------------------------
5. РЕКОМЕНДАЦИИ
   ✔ Низкая доля аварийных переходов (<1%). Система стабильна. Продолжайте мониторинг.
```

**Примечания**  
- Если за период нет ни одного перехода, отчёт укажет на отсутствие данных.
- Аварийные состояния определяются жёстко: `correlation < 0` и `os_trend = -1`.  
- В разделе динамики используются сырые данные из `transition_log` без учёта забывания.

## 3. mchain_summary_report

**Назначение**  
Сводный отчёт, объединяющий информацию о достоверности прогнозов, анализе аварийных переходов, текущей конфигурации модели и состоянии системы. Позволяет быстро оценить «здоровье» всей цепи Маркова.

**Сигнатура**  
```sql
mchain_summary_report(
    p_start TIMESTAMPTZ DEFAULT NULL,
    p_end   TIMESTAMPTZ DEFAULT NULL
)
RETURNS TEXT
```

**Входные параметры**  
Аналогичны `mchain_incident_transitions_report`:  
- `p_start` – начало периода анализа аварийных переходов (по умолч. 7 дней назад).  
- `p_end` – конец периода (по умолч. текущий момент).

**Возвращаемое значение**  
Текст, включающий следующие блоки:

1. **Конфигурация модели**  
   - Флаг адаптивного забывания, режим `alpha` (фиксированный или динамический), значения `base_alpha`, `min_alpha`, `half_life`, интервал забывания, порог `min_transitions_for_forgetting`, глубина хранения журнала, времена последнего забывания и последнего инцидента.

2. **Текущее состояние системы**  
   - `state_id` (вычисленный через `mchain_get_current_state_id`), его параметры (`correlation`, `os_trend`, `wait_trend`).  
   - Прогноз риска на следующую минуту (вызов `mchain_predict_risk_1min`).

3. **Отчёт о достоверности прогнозов** (полный вывод `mchain_reliability_report`).

4. **Анализ переходов в аварию за период** (полный вывод `mchain_incident_transitions_report` с переданными `p_start`/`p_end`).

5. **Общие рекомендации**  
   - На основе рейтинга достоверности (из п.3) даётся совет: «не полагаться на прогнозы», «использовать с осторожностью» или «модель надёжна».  
   - Если адаптивное забывание выключено – рекомендация включить.  
   - Если давно не было инцидентов (>30 дней) и используется адаптивный `alpha` – совет выполнить ручную коррекцию.

**Логика работы**

- Извлекает текущую конфигурацию из `markov_config`.
- Вызывает `mchain_get_current_state_id` и `mchain_predict_risk_1min` (ошибки перехватываются).
- Вызывает `mchain_reliability_report()` и `mchain_incident_transitions_report(p_start, p_end)`.
- Парсит числовой рейтинг из текста отчёта достоверности (регулярное выражение) для формирования итоговой рекомендации.
- Собирает все блоки в один текст с разделителями.

**Пример вызова**
```sql
-- Сводный отчёт за последние 30 дней
SELECT mchain_summary_report(now() - interval '30 days', now());

-- Без параметров (период аварий = 7 дней)
SELECT mchain_summary_report();
```

**Пример вывода (структура)**
```
СВОДНЫЙ ОТЧЁТ ПО СОСТОЯНИЮ ЦЕПИ МАРКОВА
====================================================================

Период анализа переходов: 2025-05-25 10:00:00 – 2025-06-01 10:00:00
Дата формирования отчёта: 2025-06-01 10:05:22

--------------------------------------------------------------------
1. КОНФИГУРАЦИЯ МОДЕЛИ
   Адаптивное забывание: ВКЛ
   Динамический alpha: ВКЛ
   base_alpha: 0.1
   min_alpha: 0.01
   half_life (дни): 7
   ...

--------------------------------------------------------------------
2. ТЕКУЩЕЕ СОСТОЯНИЕ СИСТЕМЫ
   State ID: 95
   Параметры: correlation=0.2, os_trend=0, wait_trend=1
   Прогноз риска на следующую минуту: 0.03 (risk_calculated)

--------------------------------------------------------------------
3. ДОСТОВЕРНОСТЬ ПРОГНОЗОВ
   ... (полный отчёт mchain_reliability_report) ...

--------------------------------------------------------------------
4. АНАЛИЗ ПЕРЕХОДОВ В АВАРИЮ ЗА ПЕРИОД
   ... (полный отчёт mchain_incident_transitions_report) ...

====================================================================
5. ОБЩИЕ РЕКОМЕНДАЦИИ
   ✔ Модель имеет удовлетворительную достоверность (рейтинг 4). Прогнозы можно использовать с осторожностью.
   → Адаптивное забывание включено. Рекомендуется следить за стабильностью.
```

**Примечания**  
- Отчёт может быть очень большим из-за включения двух других отчётов.  
- При ошибках в вызове подотчётов в соответствующих блоках выводится сообщение об ошибке, а не прерывается выполнение.

## 4. mchain_incident_state_detail_report

**Назначение**  
Предоставляет детальную информацию **по каждому аварийному состоянию**, в которое были переходы за указанный период. Для каждого такого состояния показывает:

- Сырую частоту переходов в него (из `transition_log`).
- Текущий вес в модели (из `markov_frequencies`, с учётом забывания).
- Топ-3 предшествующих состояний (из каких состояний чаще всего приходили в данное аварийное состояние) с количеством переходов и процентной долей.

Полезна для выявления «слабых мест» – конкретных аварийных состояний, которые возникают слишком часто или неожиданно.

**Сигнатура**  
```sql
mchain_incident_state_detail_report(
    p_start TIMESTAMPTZ DEFAULT NULL,
    p_end   TIMESTAMPTZ DEFAULT NULL
)
RETURNS TEXT
```

**Входные параметры**  
Аналогичны предыдущим отчётам:  
- `p_start` – начало периода (по умолч. 7 дней назад).  
- `p_end` – конец периода (по умолч. текущий момент).

**Возвращаемое значение**  
Текстовый отчёт, где для каждого аварийного состояния (`correlation < 0 AND os_trend = -1`), **имевшего хотя бы один переход за период**, выводятся:

- Идентификатор состояния (`state_id`).
- Значение корреляции и тренд ожиданий (`wait_trend`).
- Сырое количество переходов в это состояние за период.
- Вес в модели `markov_frequencies`.
- Список до 3 предшествующих состояний с их параметрами, количеством переходов и долей (в процентах от всех переходов в данное аварийное состояние).

Если ни одного перехода в аварийные состояния не было – выводится соответствующее сообщение.

**Логика работы**

1. Получает список всех аварийных состояний.
2. Фильтрует только те, для которых `EXISTS` запись в `transition_log` за период.
3. Для каждого такого состояния:
   - Считает `COUNT(*)` из `transition_log`.
   - Получает `frequency` из `markov_frequencies` (первое попавшееся, т.к. в этой таблице может быть несколько строк с одинаковым `to_state`? На самом деле `(from_state, to_state)` – составной ключ, поэтому для данного `to_state` может быть много строк с разными `from_state`. Функция берёт `LIMIT 1` – это **некорректно**: вес модели для состояния как суммы частот всех входящих переходов. Однако в текущей реализации используется `SELECT frequency FROM markov_frequencies WHERE to_state = ... LIMIT 1`, что возвращает произвольную строку. **Это баг**, но в рамках документации описываем поведение как есть (или можно отметить как ограничение).  
   - Для топ-3 предшественников: группирует `transition_log` по `from_state`, считает количество переходов, вычисляет процент от общего числа переходов в это состояние, берёт первые 3.
4. Формирует текстовый отчёт.

**Пример вызова**
```sql
-- Детали по аварийным состояниям за последние 30 дней
SELECT mchain_incident_state_detail_report(now() - interval '30 days', now());
```

**Пример вывода (фрагмент)**
```
ДЕТАЛЬНЫЙ ОТЧЁТ ПО АВАРИЙНЫМ СОСТОЯНИЯМ (только с переходами за период)
================================================================================

Период анализа: 2025-05-01 00:00:00 – 2025-06-01 00:00:00
Аварийных состояний с переходами за период: 3

--------------------------------------------------------------------------------

Состояние #42
   Корреляция: -0.5
   Тренд ожиданий (wait_trend): 1
   Сырых переходов за период: 23
   Вес в модели (с забыванием): 12.45
   Топ-3 предшествующих состояния (по переходам за период):
      -> #15 (r=0.3, os=-1, w=0): 12 переходов (52.2%)
      -> #87 (r=-0.2, os=0, w=1): 6 переходов (26.1%)
      -> #102 (r=0.1, os=1, w=-1): 3 перехода (13.0%)

--------------------------------------------------------------------------------

Состояние #78
   ...
```

**Ограничения**  
- Поле «Вес в модели» в текущей реализации получается через `LIMIT 1`, что не отражает суммарный вес всех входящих переходов в данное состояние. Для корректного анализа следует модифицировать функцию: использовать `SUM(frequency) WHERE to_state = ...`. В документации стоит отметить этот нюанс.

**Примечания**  
- Отображаются только аварийные состояния, в которые **были реальные переходы** за период. Это уменьшает объём отчёта.
- Топ предшественников строится на основе **сырых данных**, а не взвешенных частот из модели.

## 5. mchain_health_check

**Назначение**  
Выполняет быструю диагностику состояния цепи Маркова и возвращает статус (OK, WARNING, CRITICAL) с поясняющим сообщением. Предназначена для интеграции с системами мониторинга (например, Zabbix, Prometheus, pgAgent).

**Сигнатура**  
```sql
mchain_health_check()
RETURNS TABLE (status TEXT, message TEXT)
```

**Входные параметры**  
Нет.

**Возвращаемое значение**  
Таблица из одной строки с двумя колонками:

- `status` – одно из значений: `'OK'`, `'WARNING'`, `'CRITICAL'`.
- `message` – текстовое сообщение, описывающее проблему или подтверждающее нормальную работу.

**Логика работы (последовательность проверок)**

| № | Проверка | Условие для WARNING | Условие для CRITICAL |
|---|----------|---------------------|----------------------|
| 1 | Достоверность прогнозов (рейтинг 0–5) | рейтинг 1–2 | рейтинг = 0 |
| 2 | Рост аварийных переходов (последние 7 дней vs предыдущие 7) | рост > 2× | рост > 3× |
| 3 | Забывание: давность последнего вызова `mchain_apply_forgetting` | `last_forget_time < now() - 2*interval_minute` | (только WARNING) |
| 4 | Активность: есть ли переходы за последние 10 минут | – | отсутствие переходов |
| 5 | Наличие данных в `markov_frequencies` | таблица пуста | (только WARNING) |

Каждая проверка может повысить строгость статуса (CRITICAL важнее WARNING, WARNING важнее OK). Сообщения накапливаются через разделитель `'; '`.

Если все проверки пройдены, возвращается `status = 'OK'` и сообщение `'Все проверки пройдены. Цепь Маркова работает штатно.'`.

**Примеры вызова и вывода**

```sql
-- Запуск проверки
SELECT * FROM mchain_health_check();
```

**Пример 1 – нормальная работа**
```
 status | message
--------+---------------------------------------------------------
 OK     | Все проверки пройдены. Цепь Маркова работает штатно.
```

**Пример 2 – предупреждение**
```
 status  | message
---------+---------------------------------------------------------
 WARNING | Низкая достоверность (2). Требуется больше данных.; Значительный рост аварийных переходов (0.5% -> 1.8%)
```

**Пример 3 – критическое состояние**
```
 status   | message
----------+---------------------------------------------------------
 CRITICAL | Модель не обучена (рейтинг 0). Нет данных или менее 100 переходов.; Нет переходов за последние 10 минут (возможно, остановлен mchain_train_step).
```

**Примечания**  
- Функция **не изменяет** данные, только читает таблицы.  
- Для корректной работы требуется наличие таблиц `markov_config`, `transition_log`, `markov_frequencies` и заполненного `state_descriptions`.  
- Проверка роста аварийных переходов требует минимум 14 дней данных; при недостатке данных эта проверка пропускается.  
- Рекомендуется вызывать `mchain_health_check()` с периодичностью 5–10 минут через внешний мониторинг.

## Заключение

Представленные функции предоставляют полный набор инструментов для:

- **Оценки качества модели** – `mchain_reliability_report`.
- **Анализа аварийных переходов** – `mchain_incident_transitions_report` (общий) и `mchain_incident_state_detail_report` (детальный).
- **Общего мониторинга** – `mchain_summary_report` (сводный) и `mchain_health_check` (для автоматических систем).

Используйте их регулярно для поддержания надёжности прогнозов и своевременной настройки параметров забывания.
