## Таблица `forget_log`

### 1. Назначение

Таблица `forget_log` служит **журналом форсированных (экстренных) забываний** в системе адаптивной цепи Маркова. Она фиксирует каждый случай принудительного снижения весов исторических переходов, происходящего по решению процедуры `check_and_forget()` или при вызове `emergency_forget()`. Журнал позволяет:

- Анализировать причины, по которым модель была вынуждена «забыть» часть истории (дрейф распределений, ухудшение точности прогнозов, внешние события).
- Отслеживать динамику метрик (KL-дивергенция, χ², Brier Score, отклонение операционной скорости) в момент принятия решения.
- Оценивать эффективность порогов срабатывания и частоту форсированных забываний для дальнейшей калибковки.
- Аудитировать работу механизма адаптивного забывания (включая ручные вмешательства).

Таблица не участвует непосредственно в расчётах вероятностей, а служит исключительно для мониторинга и отладки.

### 2. Определение (DDL)

```sql
DROP TABLE IF EXISTS forget_log;
CREATE TABLE forget_log (
    id            BIGSERIAL PRIMARY KEY,
    ts            TIMESTAMPTZ DEFAULT now(),
    alpha         REAL NOT NULL,
    triggered_by  TEXT[],                 -- массив сработавших признаков
    kl_div        REAL,
    chi2_val      REAL,
    brier_score   REAL,
    os_deviation  REAL,
    details       TEXT
);
COMMENT ON TABLE forget_log IS 'Журнал форсированных забываний (методика 4.1)';
```

### 3. Детальное описание колонок

| Колонка | Тип | Обязательность | Описание |
|---------|-----|----------------|-----------|
| `id` | `BIGSERIAL` | NOT NULL (PK) | Уникальный идентификатор записи, генерируется автоматически. |
| `ts` | `TIMESTAMPTZ` | DEFAULT now() | Время применения забывания (с точностью до часового пояса). Используется для хронологического анализа и очистки по сроку хранения. |
| `alpha` | `REAL` | NOT NULL | Фактически применённый коэффициент забывания (обычно в диапазоне 0.0–0.5). Частоты переходов умножаются на `(1 – alpha)`. Значение может суммироваться из нескольких признаков, но ограничивается конфигурационным параметром `forget_alpha_max`. |
| `triggered_by` | `TEXT[]` | NULL | Массив строк, перечисляющих признаки, которые в совокупности вызвали забывание. Возможные значения: `'KL'`, `'Chi2'`, `'OS'`, `'Brier'`, `'Infra'`, `'Diurnal'`. Каждый признак добавляет свой вклад в `alpha`. |
| `kl_div` | `REAL` | NULL | Значение KL-дивергенции между распределением состояний за последний час и эталонным распределением (рассчитанное функцией `calculate_kl_divergence()`). Используется для обнаружения дрейфа распределения. |
| `chi2_val` | `REAL` | NULL | Значение χ²-критерия (хи-квадрат) между наблюдаемыми частотами состояний за последний час и ожидаемыми из эталона. Рассчитывается функцией `calculate_chi_squared()`. |
| `brier_score` | `REAL` | NULL | Средний Brier Score прогнозов за последние 2 часа (вычисляется на основе таблицы `forecast_log`). Характеризует калибровку вероятностных прогнозов модели. |
| `os_deviation` | `REAL` | NULL | Относительное отклонение текущей операционной скорости (SMA20) от исторического среднего для данного часа суток. Рассчитывается функцией `get_os_deviation()`. |
| `details` | `TEXT` | NULL | Дополнительная текстовая информация, формируемая процедурой `check_and_forget()`: содержит строку с численными значениями сработавших метрик (например, `'KL=0.52; Chi2=245.3; Brier=0.31;'`). |

### 4. Связи с другими объектами

- **Таблицы конфигурации** – `markov_config` хранит пороговые значения (`kl_threshold`, `chi2_threshold`, `os_dev_threshold`, `brier_threshold`), максимальный alpha (`forget_alpha_max`) и количество циклов подтверждения (`confirmation_cycles`). При превышении порогов по нескольким последовательным проверкам инициируется запись в `forget_log`.
- **Функция `check_and_forget()`** – основной генератор записей. Она вычисляет все метрики, определяет сработавшие признаки, формирует `alpha` и вставляет строку.
- **Функция `emergency_forget()`** – вставляет запись в `forget_log` с массивом `triggered_by = ['Emergency_<event_type>']` и заданным пользователем `alpha` (при ручном или автоматическом внешнем вмешательстве).
- **Функция `clean_forget_log()`** – удаляет записи старше заданного в `markov_config.forget_log_retention_days` срока (по умолчанию 90 дней). Вызывается по расписанию (например, ежемесячно).
- **Функция `get_forget_log()`** – предоставляет удобный интерфейс для чтения журнала в виде массива строк за указанный период.

### 5. Логика заполнения (из `check_and_forget()`)

Процедура `check_and_forget()` выполняется по расписанию (например, каждые 15 минут). Её упрощённый алгоритм применительно к `forget_log`:

1. Проверяет, включено ли адаптивное забывание (`adaptive_forgetting_enabled = true`).
2. Вычисляет метрики:
   - `kl_val` – KL-дивергенция за 60 мин.
   - `chi2_val` – χ² за 60 мин.
   - `os_dev` – отклонение скорости.
   - `brier_val` – Brier Score за 2 часа.
   - `infra_flag` – наличие неподтверждённых внешних событий (деплой, сбой).
   - `diurnal_flag` – признак внутридневного дрейфа.
3. Сохраняет флаги в `check_state` и подсчитывает, сколько последних циклов подряд (не менее `confirmation_cycles`) был хотя бы один признак.
4. Если условие подтверждения выполнено, суммирует вклад каждого признака в `alpha_eff`:
   - `KL`, `Chi2`, `OS`, `Brier` → +0.1
   - `Diurnal` → +0.02
   - `Infra` → `alpha_eff = max(alpha_eff, 0.3)`
5. Ограничивает `alpha_eff` значением `forget_alpha_max` (по умолчанию 0.5).
6. Вызывает `apply_forgetting(alpha_eff)` для фактического изменения частот.
7. Вставляет запись в `forget_log` с указанием:
   - `alpha = alpha_eff`
   - `triggered_by` – массив сработавших признаков
   - `kl_div`, `chi2_val`, `brier_score`, `os_deviation` – фактические значения метрик
   - `details` – текстовая строка с этими значениями.

### 6. Пример содержимого

| id | ts | alpha | triggered_by | kl_div | chi2_val | brier_score | os_deviation | details |
|----|----|-------|--------------|--------|----------|-------------|--------------|---------|
| 42 | 2025-05-27 14:30:00+00 | 0.3 | {KL,Chi2,Infra} | 0.52 | 245.3 | 0.27 | 0.12 | KL=0.52; Chi2=245.3; Brier=0.27; |
| 43 | 2025-05-27 09:15:00+00 | 0.1 | {OS} | NULL | NULL | NULL | 0.35 | OS_dev=0.35; |
| 44 | 2025-05-26 22:00:00+00 | 0.4 | {Emergency_deploy} | NULL | NULL | NULL | NULL | Forced by external event |

### 7. Функции для работы с таблицей

- **`clean_forget_log()`** – удаляет строки старше `forget_log_retention_days` (задаётся в `markov_config`). Рекомендуется запускать по расписанию (например, `0 4 1 * *` – 1-го числа каждого месяца в 4:00).
- **`get_forget_log(p_start, p_end)`** – возвращает массив текстовых строк с подробным описанием записей за период. Полезна для быстрой выгрузки в отчёты или скрипты.

```sql
-- Пример вызова
SELECT unnest(get_forget_log('2025-05-01', '2025-05-31'));
```

### 8. Рекомендации по использованию

- **Мониторинг частоты забываний** – если `forget_log` пополняется чаще 6 раз в сутки, стоит пересмотреть пороги в `markov_config` или увеличить `confirmation_cycles`.
- **Анализ причин** – группируйте записи по `triggered_by`, чтобы понять, какой признак чаще всего вызывает забывание (например, `OS` может указывать на проблемы с инфраструктурой).
- **Калибровка порогов** – в первые недели работы модели запускайте `check_and_forget()` в режиме мониторинга (без реального `apply_forgetting`) и анализируйте накопленные метрики, затем корректируйте `kl_threshold`, `chi2_threshold` и др.
- **Аудит ручных вмешательств** – все вызовы `emergency_forget()` автоматически логируются с типом `Emergency_*`, что позволяет отслеживать административные действия.

### 9. Очистка и хранение

- Срок хранения задаётся колонкой `forget_log_retention_days` в таблице `markov_config` (по умолчанию 90 дней).
- Удаление старых записей выполняется функцией `clean_forget_log()`.
- Рекомендуемый график очистки: раз в месяц (избегает избыточного роста таблицы, но сохраняет достаточно истории для квартального аудита).

Таким образом, `forget_log` является ключевым элементом observability механизма адаптивного забывания, позволяя инженерам по мониторингу и администраторам понимать поведение самообучающейся модели в динамично меняющейся среде.
