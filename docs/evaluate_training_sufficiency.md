# Функция `evaluate_training_sufficiency` – комплексная проверка достаточности обучения цепи Маркова

## Общая информация

| Свойство | Значение |
|----------|----------|
| **Полное имя** | `evaluate_training_sufficiency(test_start DATE, test_end DATE, model_date_old DATE, model_date_new DATE)` |
| **Язык** | PL/pgSQL |
| **Возвращаемое значение** | `TABLE(criterion TEXT, value REAL, threshold TEXT, passed BOOLEAN, details TEXT)` |
| **Назначение** | Оценка зрелости и стабильности модели цепи Маркова по четырём объективным критериям: статистическая обеспеченность, стабильность вероятностей, качество прогнозов (Brier Score) и соответствие стационарного распределения эмпирическим данным |
| **Вызов** | Еженедельно (например, в пятницу вечером) после создания снимка матрицы и накопления достаточного количества прогнозов |
| **Версия кода** | 10.0 |

Функция является **главным инструментом верификации** модели – позволяет принять решение о том, можно ли доверять прогнозам и снижать интенсивность забывания, или же требуется дальнейшее обучение и настройка.

---

## Сигнатура

```sql
CREATE OR REPLACE FUNCTION evaluate_training_sufficiency(
    test_start      DATE DEFAULT NULL,
    test_end        DATE DEFAULT NULL,
    model_date_old  DATE DEFAULT NULL,
    model_date_new  DATE DEFAULT NULL
)
RETURNS TABLE (
    criterion TEXT,
    value     REAL,
    threshold TEXT,
    passed    BOOLEAN,
    details   TEXT
)
LANGUAGE plpgsql
AS $$ ... $$;
```

**Параметры:**
- `test_start`, `test_end` – границы тестового периода (даты) для оценки качества прогнозов (критерий C3). Если `NULL`, этот критерий пропускается.
- `model_date_old`, `model_date_new` – даты обучения двух версий модели (старая и новая) для сравнения Brier Score. Если `NULL`, критерий C3 не вычисляется.

**Возвращаемый набор** – ровно 4 строки (по одной на каждый критерий), даже если какой-то критерий не удалось вычислить (соответствующие поля будут содержать пояснения).

---

## Четыре критерия достаточности обучения

| Критерий | Символ | Описание | Пороговое значение |
|----------|--------|----------|---------------------|
| **Статистическая обеспеченность** | C1 | Для состояний, встречающихся с частотой >1% от общего числа переходов, каждое должно быть наблюдено не менее 50 раз. | `value` = количество состояний с `n_i < 50`; `passed` = 0 |
| **Стабильность вероятностей** | C2 | Максимальное абсолютное изменение любой вероятности перехода между текущей матрицей и её снимком недельной давности. | `passed` = `D_max < 0.05` |
| **Качество прогнозов (Brier Score)** | C3 | Улучшение среднеквадратичной ошибки прогноза при добавлении ещё одной недели обучения. | `passed` = `improvement < 0.01` |
| **Стационарность распределения** | C4 | KL-дивергенция между стационарным распределением цепи и эмпирическим распределением состояний за последнюю неделю. | `passed` = `KL < 0.1` |

---

## Детальное описание алгоритма

### 1. Критерий C1: `n_i >= 50` для частых состояний (>1%)

**Цель:** убедиться, что модель имеет достаточно данных для надёжной оценки вероятностей переходов из часто встречающихся состояний.

**Алгоритм:**

```sql
SELECT COUNT(*) INTO total_transitions FROM transition_log;

WITH state_counts AS (
    SELECT from_state, COUNT(*) AS n_i,
           COUNT(*)::REAL / total_transitions AS freq
    FROM transition_log
    GROUP BY from_state
)
SELECT array_agg(from_state) INTO states_above_1pct
FROM state_counts
WHERE freq > 0.01;

SELECT COUNT(*) INTO n_i_fail
FROM (
    SELECT from_state, COUNT(*) AS n_i
    FROM transition_log
    WHERE from_state = ANY(states_above_1pct)
    GROUP BY from_state
) sub
WHERE n_i < 50;
```

**Выходные поля для C1:**
- `criterion` = `'C1: n_i >= 50 (для частых >1%)'`
- `value` = количество состояний с `freq > 1%`, у которых `n_i < 50` (целое число, приведённое к `REAL`)
- `threshold` = `'0'`
- `passed` = `(n_i_fail = 0)`
- `details` = строка вида `'Состояний с частотой >1%: X, из них с n_i<50: Y'`

**Особые случаи:**
- Если `transition_log` пуст → `value = 0`, `passed = FALSE`, `details = 'transition_log пуст'`.

---

### 2. Критерий C2: максимальное изменение вероятностей за две недели

**Цель:** проверить, что матрица вероятностей стабилизировалась и не претерпевает значительных изменений от недели к неделе.

**Алгоритм:**

```sql
SELECT MAX(ABS(COALESCE(p.probability,0) - COALESCE(pw.probability,0))) INTO d_max
FROM (SELECT DISTINCT from_state, to_state FROM markov_probabilities
      UNION SELECT DISTINCT from_state, to_state FROM markov_probabilities_prev_week) all_trans
LEFT JOIN markov_probabilities p USING (from_state, to_state)
LEFT JOIN markov_probabilities_prev_week pw USING (from_state, to_state);
```

**Пояснение:**
- Берётся полное объединение всех пар `(from_state, to_state)`, встречающихся хотя бы в одной из двух матриц.
- Для каждой пары вычисляется абсолютная разница вероятностей (отсутствие в одной из таблиц трактуется как 0).
- `d_max` – максимум по всем парам.

**Выходные поля для C2:**
- `criterion` = `'C2: max |P_new - P_old|'`
- `value` = вычисленное `d_max` (или `-1`, если нет данных)
- `threshold` = `'D < 0.05'`
- `passed` = `(d_max IS NOT NULL AND d_max < 0.05)`
- `details` = либо `'D_max = 0.xxxx'`, либо `'Нет исторической матрицы за прошлую неделю'`

**Зависимость:**  
Таблица `markov_probabilities_prev_week` должна быть предварительно заполнена функцией `snapshot_markov_prev_week()` (выполняется еженедельно).

---

### 3. Критерий C3: улучшение Brier Score (плато качества прогнозов)

**Цель:** оценить, перестало ли качество прогнозов значимо улучшаться при добавлении новых данных. Если улучшение меньше 0.01 – обучение достаточно.

**Алгоритм:**

Если все четыре параметра (`test_start`, `test_end`, `model_date_old`, `model_date_new`) переданы и не `NULL`:

```sql
SELECT bs_improvement, sufficient INTO brier_change, passed
FROM compare_brier_scores(test_start, test_end, model_date_old, model_date_new);
```

Функция `compare_brier_scores` вычисляет Brier Score для двух моделей на общем тестовом периоде и возвращает:
- `older_bs`, `newer_bs`
- `bs_improvement = GREATEST(older_bs - newer_bs, 0.0)`
- `sufficient = (older_bs - newer_bs) < 0.01`

**Если параметры не заданы** – критерий пропускается:

```sql
criterion := 'C3: Brier Score изменение < 0.01';
value := -1;
threshold := '< 0.01';
passed := FALSE;
details := 'Не заданы параметры тестового периода/моделей. Выполните еженедельное сравнение вручную.';
```

**Выходные поля для C3:**
- `criterion` = `'C3: Brier Score изменение < 0.01'`
- `value` = `bs_improvement` (или `-1`)
- `threshold` = `'< 0.01'`
- `passed` = результат `sufficient` или `FALSE` при отсутствии параметров
- `details` = пояснение (`'Изменение BS = 0.xxx'` или сообщение о необходимости ручного запуска)

**Важное требование:**  
Перед вызовом `evaluate_training_sufficiency` с параметрами C3 таблица `forecast_log` должна быть заполнена прогнозами для **обеих сравниваемых моделей** на тестовом периоде. Это достигается запуском `log_forecast` для каждой версии модели (см. сценарий 2 в описании `log_forecast`).

---

### 4. Критерий C4: KL-дивергенция стационарного и эмпирического распределений

**Цель:** проверить, что стационарное распределение цепи Маркова (долгосрочный прогноз) согласуется с реальным распределением состояний, наблюдённым за последнюю неделю. Расхождение `KL < 0.1` говорит о том, что модель адекватно отражает стационарный режим.

**Алгоритм:**

Вызывает специализированную функцию `check_kl_divergence()`, которая:

1. Вычисляет стационарное распределение `π` через `get_stationary_distribution()`.
2. Подсчитывает эмпирические частоты состояний за последние 7 дней из `transition_log`.
3. Вычисляет `KL = Σ π_i * ln(π_i / emp_i)` (только для состояний, где оба распределения ненулевые).
4. Возвращает `(kl_value, threshold=0.1, passed=kl_value < 0.1)`.

**Выходные поля для C4:**
- `criterion` = `'C4: KL(pi || emp) < 0.1'`
- `value` = `kl_value` (или `NULL` → `-1` в выводе?)
- `threshold` = `'< 0.1'`
- `passed` = результат `check_kl_divergence().passed`
- `details` = `'KL = 0.xxxx'` или `'Нет данных за последнюю неделю'`

**Примечание:** Если за последние 7 дней в `transition_log` нет записей, `kl_value` возвращается `NULL`, `passed = FALSE`.

---

## Формат выходных данных

Функция возвращает **4 строки** – по одной на каждый критерий. Столбцы:

| Столбец | Тип | Описание |
|---------|-----|----------|
| `criterion` | `TEXT` | Название критерия (C1…C4) с кратким описанием порога |
| `value` | `REAL` | Вычисленное значение (количество нарушений, максимальная разница, улучшение BS, KL-дивергенция). Для C3 при отсутствии параметров – `-1`. |
| `threshold` | `TEXT` | Строковое представление порога (например `'0'`, `'D < 0.05'`, `'< 0.01'`, `'< 0.1'`) |
| `passed` | `BOOLEAN` | `TRUE`, если значение соответствует порогу (или `FALSE` для невычисленного C3) |
| `details` | `TEXT` | Дополнительная информация: количество состояний, конкретное значение `D_max`, `BS_improvement`, `KL_value` или сообщение об отсутствии данных |

## Примеры вызова и интерпретации

### Пример 1. Полная проверка с заданием всех параметров

```sql
SELECT * FROM evaluate_training_sufficiency(
    test_start     => '2025-05-19',
    test_end       => '2025-05-20',
    model_date_old => '2025-05-12',
    model_date_new => '2025-05-19'
);
```

**Результат:**

| criterion | value | threshold | passed | details |
|-----------|-------|-----------|--------|---------|
| C1: n_i >= 50 (для частых >1%) | 0 | 0 | true | Состояний с частотой >1%: 42, из них с n_i<50: 0 |
| C2: max \|P_new - P_old\| | 0.032 | D < 0.05 | true | D_max = 0.0321 |
| C3: Brier Score изменение < 0.01 | 0.007 | < 0.01 | true | Изменение BS = 0.0073 |
| C4: KL(pi \|\| emp) < 0.1 | 0.085 | < 0.1 | true | KL = 0.0852 |

**Интерпретация:** Все четыре критерия выполнены → модель достаточно обучена, можно доверять прогнозам и снижать интенсивность забывания (уменьшить `alpha`).

---

### Пример 2. Недостаточно данных (частые состояния имеют мало наблюдений)

```sql
SELECT * FROM evaluate_training_sufficiency();  -- без параметров для C3
```

**Результат:**

| criterion | value | threshold | passed | details |
|-----------|-------|-----------|--------|---------|
| C1: n_i >= 50 (для частых >1%) | 3 | 0 | false | Состояний с частотой >1%: 15, из них с n_i<50: 3 |
| C2: max \|P_new - P_old\| | 0.092 | D < 0.05 | false | D_max = 0.0916 |
| C3: Brier Score изменение < 0.01 | -1 | < 0.01 | false | Не заданы параметры тестового периода/моделей. Выполните еженедельное сравнение вручную. |
| C4: KL(pi \|\| emp) < 0.1 | 0.247 | < 0.1 | false | KL = 0.247 |

**Интерпретация:** Модель нестабильна, частоты отдельных состояний недостаточны, вероятности сильно меняются, стационарное распределение расходится с эмпирикой. Необходимо продолжить сбор данных и, возможно, увеличить `alpha` для более быстрой адаптации.

---

### Пример 3. Отсутствие исторической матрицы за прошлую неделю

**Результат (фрагмент):**

| criterion | value | threshold | passed | details |
|-----------|-------|-----------|--------|---------|
| C2: max \|P_new - P_old\| | -1 | D < 0.05 | false | Нет исторической матрицы за прошлую неделю |

**Решение:** Запланировать еженедельный вызов `snapshot_markov_prev_week()`.

## Зависимости от других функций и таблиц

| Объект | Тип | Назначение | Используется в критерии |
|--------|-----|------------|--------------------------|
| `transition_log` | Таблица | Сырые переходы | C1, C4 (через `check_kl_divergence`) |

| `check_kl_divergence()` | Функция | Расчёт KL-дивергенции | C4 |
| `get_stationary_distribution()` | Функция | Вычисление стационарного распределения | внутри `check_kl_divergence` |

## Рекомендации по эксплуатации

### 1. Регулярный запуск
Настройте еженедельное выполнение в пятницу вечером (после завершения сбора данных за неделю и после создания снимка матрицы):

```sql
-- В 18:10 каждую пятницу
SELECT cron.schedule('check-sufficiency', '10 18 * * 5', $$
    SELECT evaluate_training_sufficiency(
        test_start     => (CURRENT_DATE - INTERVAL '2 days')::DATE,
        test_end       => (CURRENT_DATE - INTERVAL '1 day')::DATE,
        model_date_old => (CURRENT_DATE - INTERVAL '7 days')::DATE,
        model_date_new => CURRENT_DATE
    );
$$);
```

### 2. Логирование результатов
Создайте отдельную таблицу для хранения истории проверок:

```sql
CREATE TABLE training_sufficiency_log (
    check_date TIMESTAMPTZ PRIMARY KEY,
    c1_passed BOOLEAN, c1_value REAL, c1_details TEXT,
    c2_passed BOOLEAN, c2_value REAL, c2_details TEXT,
    c3_passed BOOLEAN, c3_value REAL, c3_details TEXT,
    c4_passed BOOLEAN, c4_value REAL, c4_details TEXT
);
```

И вставляйте результаты после каждого вызова.

### 3. Действия при нарушении критериев

| Нарушенный критерий | Возможные причины | Рекомендуемые действия |
|---------------------|-------------------|-------------------------|
| **C1** | Недостаточно исторических данных | Увеличить период сбора данных; проверить, что `transition_log` заполняется каждую минуту |
| **C2** | Вероятности ещё не стабилизировались | Продолжить обучение без изменений; возможно, увеличить `alpha` для ускорения адаптации |
| **C3** | Качество прогнозов продолжает заметно улучшаться | Добавить ещё одну неделю обучения перед повторной проверкой |
| **C4** | Изменился профиль нагрузки (дрейф) | Пересмотреть механизм забывания; возможно, обучить отдельные матрицы для разных часов или дней недели |

### 4. Когда можно снижать интенсивность забывания
Если все четыре критерия выполняются в течение нескольких недель подряд, можно:

```sql
UPDATE markov_config SET alpha = 0.005, interval_hours = 2;
```

И продолжать мониторинг. При повторном появлении нарушений – вернуть `alpha` к 0.01 и уменьшить интервал до 1 часа.

## Ограничения и предположения

- **Критерий C3 требует предварительного заполнения `forecast_log` для обеих моделей на тестовом периоде.**  
  Без этого он не может быть вычислен, и функция вернёт `passed = false` с пояснением.  
  **Решение:** реализовать сценарий 2 из описания `log_forecast`: прогон исторических данных через архивные матрицы.

- **Критерий C2 требует наличия таблицы `markov_probabilities_prev_week`.**  
  Если снимок не создавался, критерий не может быть выполнен.

- **Критерий C4 использует `get_stationary_distribution()`, которая может сходиться медленно для плохо обусловленных матриц.**  
  Для 189 состояний сходимость обычно достигается за 100–200 итераций, но при разреженных матрицах может потребоваться больше. Функция использует `max_iter = 1000` и `tol = 1e-6`, что достаточно для практических целей.

- **Все четыре критерия независимы** – функция не делает общий вывод "обучение достаточно", оставляя его на усмотрение аналитика. Рекомендуется считать обучение достаточным, когда **все четыре** `passed = true`.

---

## Возможные расширения

1. **Добавление пятого критерия – калибровка прогнозов** (например, тест Хосмера–Лемешоу). Потребует изменений в `forecast_log` и дополнительной функции.

2. **Параметризация порогов** – вынести пороговые значения (50, 0.05, 0.01, 0.1) в таблицу конфигурации `markov_config_thresholds`.

3. **Возврат общего вердикта** – добавить выходной параметр `overall_sufficient BOOLEAN`, вычисляемый как `AND` всех `passed`.

4. **Учёт взвешивания состояний в C1** – вместо абсолютного порога 50 использовать формулу, основанную на дисперсии оценки вероятности.

---

## Заключение

Функция `evaluate_training_sufficiency` предоставляет **объективную, количественную оценку** готовности модели цепи Маркова к практическому использованию. Она объединяет статистические критерии (C1), стабильность (C2), качество прогнозов (C3) и стационарность (C4). Регулярное применение этой функции позволяет:

- Обоснованно решать, когда можно уменьшать интенсивность забывания.
- Обнаруживать дрейф характеристик системы.
- Сравнивать разные версии модели.
- Автоматизировать процесс мониторинга качества модели в продуктивной среде.

**Рекомендуемый цикл:** еженедельный запуск → анализ результатов → корректировка параметров `markov_config` при необходимости → архивация результатов.
