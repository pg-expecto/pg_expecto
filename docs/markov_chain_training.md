# Функция `markov_chain_training` – ядро обучения цепи Маркова

## Общая информация

| Свойство | Значение |
|----------|----------|
| **Полное имя** | `markov_chain_training()` |
| **Язык** | PL/pgSQL |
| **Возвращаемое значение** | `void` |
| **Назначение** | Ежеминутное обучение цепи Маркова: сбор метрик, сдвиг состояния, прогнозирование риска, логирование, обновление частот переходов и плановое забывание |
| **Вызов** | Планировщик (pg_cron, системный cron) каждую минуту в рабочие часы |
| **Версия кода** | 10.0 |

Функция является **основным оркестратором** всего процесса адаптивного обучения марковской модели производительности PostgreSQL.

---

## Сигнатура

```sql
CREATE OR REPLACE FUNCTION markov_chain_training()
RETURNS void
LANGUAGE plpgsql
AS $$
-- тело функции
$$;
```

---

## Алгоритм работы (пошагово)

### Шаг 0. Проверка рабочего времени

Функция выполняется **только в рабочие часы**, чтобы не обучаться на нерепрезентативных данных ночью и в выходные.

```sql
IF EXTRACT(DOW FROM now()) IN (0, 6) THEN
    RAISE NOTICE 'Обучение пропущено: выходной день (%)', to_char(now(), 'Day');
    RETURN;
END IF;

IF EXTRACT(HOUR FROM now()) < 8 OR EXTRACT(HOUR FROM now()) >= 19 THEN
    RAISE NOTICE 'Обучение пропущено: нерабочее время (%)', to_char(now(), 'HH24:MI');
    RETURN;
END IF;
```

**Условия:**
- День недели: Понедельник – Пятница (1–5). Суббота (6) и воскресенье (0) пропускаются.
- Часы: с 08:00 до 19:00 (19:00 включительно не входит).

**Логирование:** при пропуске в лог PostgreSQL отправляется уведомление (`RAISE NOTICE`), но ошибка не генерируется.

---

### Шаг 1. Плановое забывание (экспоненциальное затухание частот)

Читает настройки забывания из таблицы `markov_config`.

```sql
SELECT last_forget_time, alpha, MAKE_INTERVAL(hours => interval_hours)
INTO last_forget, forget_alpha, forget_interval
FROM markov_config LIMIT 1;
```

Если с момента последнего забывания прошло `interval_hours` часов (или больше):

```sql
IF now() - last_forget >= forget_interval THEN
    PERFORM apply_forgetting(forget_alpha);
    UPDATE markov_config SET last_forget_time = now();
END IF;
```

**Что делает `apply_forgetting(alpha)`:**
- Умножает все частоты в `markov_frequencies` на `(1 - alpha)` – старые наблюдения теряют вес.
- Удаляет записи с частотой < 1e-6 (пренебрежимо малые).
- Пересчитывает матрицу вероятностей (`update_markov_probabilities()`), которая также перестраивает поглощающую матрицу.

**Параметры по умолчанию** (из `markov_config`):
- `alpha = 0.01`
- `interval_hours = 1` (забывание раз в час)

**Рекомендации по настройке:**
- Для стабильной нагрузки → `alpha = 0.01`, интервал 1 час.
- Для медленно меняющейся нагрузки → `alpha = 0.005`, интервал 2–4 часа.
- При обнаружении дрейфа → временно увеличить `alpha` или уменьшить интервал.

---

### Шаг 2. Получение текущих метрик производительности

Вызывает функцию `get_current_os_waiting_correlation_for_markov_chain()`, которая возвращает три значения:

```sql
SELECT * INTO new_values_rec
FROM get_current_os_waiting_correlation_for_markov_chain();

new_correlation := new_values_rec.current_correlation;
new_os_trend    := new_values_rec.current_os_trend;
new_wait_trend  := new_values_rec.current_wait_trend;
```

**Что возвращает эта функция:**
- `current_correlation` (REAL) – коэффициент корреляции Пирсона между операционной скоростью и ожиданиями за последний час.
- `current_os_trend` (SMALLINT) – тренд скорости (-1, 0, +1), вычисленный по углу наклона линии регрессии.
- `current_wait_trend` (SMALLINT) – тренд ожиданий (-1, 0, +1).

**Источник данных:** таблица `cluster_stat_median` (предполагается, что она содержит метрики PostgreSQL с минутным разрешением).

---

### Шаг 3. Инициализация состояния (первый запуск)

Функция всегда поддерживает **ровно одну строку** в таблице `markov_chain`.

```sql
SELECT * INTO markov_chain_rec FROM markov_chain;
```

Если `prev_correlation IS NULL` (таблица пуста или строка не инициализирована), то:

```sql
INSERT INTO markov_chain (
    prev_correlation, prev_os_trend, prev_wait_trend,
    curr_correlation, curr_os_trend, curr_wait_trend
) VALUES (
    new_correlation, new_os_trend, new_wait_trend,
    new_correlation, new_os_trend, new_wait_trend
);
RETURN;
```

**Логика:** При первом вызове нет предыдущего состояния, поэтому просто сохраняем текущие метрики и завершаемся. Переход будет зафиксирован на следующей минуте.

---

### Шаг 4. Сдвиг состояния (обновление скользящего окна)

```sql
UPDATE markov_chain
SET prev_correlation = markov_chain_rec.curr_correlation,
    prev_os_trend    = markov_chain_rec.curr_os_trend,
    prev_wait_trend  = markov_chain_rec.curr_wait_trend,
    curr_correlation = new_correlation,
    curr_os_trend    = new_os_trend,
    curr_wait_trend  = new_wait_trend;
```

Теперь:
- `prev_*` – состояние на предыдущей минуте (только что ставшее прошлым).
- `curr_*` – текущее состояние (только что измеренное).

---

### Шаг 5. Идентификация числовых кодов состояний

Использует чистую детерминированную функцию `get_state_id(r, os_trend, wait_trend)`:

```sql
prev_state := get_state_id(markov_chain_rec.prev_correlation,
                           markov_chain_rec.prev_os_trend,
                           markov_chain_rec.prev_wait_trend);
curr_state := get_state_id(markov_chain_rec.curr_correlation,
                           markov_chain_rec.curr_os_trend,
                           markov_chain_rec.curr_wait_trend);
```

Функция `get_state_id` реализована на SQL и помечена `IMMUTABLE`, что позволяет эффективно использовать индексы.

**Формула кодирования:**  
`state_id = correlation_index * 9 + (os_trend+1)*3 + (wait_trend+1)`,  
где `correlation_index = round((r + 1.0) / 0.1)` (0…20).

---

### Шаг 6. Прогнозирование риска (из предыдущего состояния)

Вычисляет вероятность перехода в **аварийную зону** на следующей минуте.

**Определение аварийной зоны:**  
`correlation < 0 AND os_trend = -1` (отрицательная корреляция и падение операционной скорости).

```sql
SELECT COALESCE(SUM(probability), 0.0) INTO risk_pred
FROM markov_probabilities
WHERE from_state = prev_state
  AND to_state IN (
      SELECT state_id FROM state_descriptions
      WHERE correlation < 0 AND os_trend = -1
  );
```

- Если сумма `NULL` (нет записей для `prev_state`) – `risk_pred = 0.0`.
- Вероятность извлекается из актуальной матрицы `markov_probabilities`, которая обновляется после каждого забывания.

---

### Шаг 7. Определение фактического исхода

Проверяет, является ли **текущее состояние** `curr_state` аварийным.

```sql
SELECT CASE WHEN correlation < 0 AND os_trend = -1 THEN 1 ELSE 0 END INTO actual
FROM state_descriptions
WHERE state_id = curr_state;
```

`actual` принимает значение:
- `1` – система вошла в аварийное состояние на этой минуте.
- `0` – состояние не аварийное.

---

### Шаг 8. Логирование прогноза

Сохраняет запись в таблице `forecast_log`:

```sql
INSERT INTO forecast_log (ts, model_train_date, predicted_risk, actual_risk, from_state, to_state)
VALUES (now(), current_date, risk_pred, actual, prev_state, curr_state);
```

**Поля:**
- `ts` – время фиксации факта (текущий момент).
- `model_train_date` – дата, до которой обучена модель (здесь `current_date`; может быть изменено для сравнения версий).
- `predicted_risk` – вероятность, предсказанная минуту назад.
- `actual_risk` – фактический исход (0 или 1).
- `from_state`, `to_state` – коды исходного и целевого состояний для детального анализа.

Этот журнал позже используется для расчёта Brier Score, калибровочных кривых и критерия достаточности обучения (C3).

---

### Шаг 9. Обновление матрицы частот переходов

Фиксирует переход из предыдущего состояния в текущее в сырой частотной таблице `markov_frequencies`.  
Вызывает специальную функцию `log_transition_and_update`:

```sql
PERFORM log_transition_and_update(
    markov_chain_rec.prev_correlation,
    markov_chain_rec.prev_os_trend,
    markov_chain_rec.prev_wait_trend,
    markov_chain_rec.curr_correlation,
    markov_chain_rec.curr_os_trend,
    markov_chain_rec.curr_wait_trend
);
```

**Что делает `log_transition_and_update`**:
- Вставляет запись в `transition_log` (с меткой времени `now()`).
- Обновляет `markov_frequencies`: либо увеличивает существующую частоту на 1, либо вставляет новую со значением 1.0.

---

### Шаг 10. Завершение

Функция завершается, возвращая `void`. Все изменения фиксируются (или откатываются) в рамках внешней транзакции (обычно автокоммит, если вызов через планировщик).

## Используемые таблицы

| Таблица | Роль в функции |
|---------|----------------|
| `markov_config` | Чтение параметров забывания (`last_forget_time`, `alpha`, `interval_hours`) |
| `markov_chain` | Хранит одну строку с предыдущим и текущим состоянием (обновляется каждый вызов) |
| `markov_probabilities` | Используется для прогноза риска (чтение) |
| `state_descriptions` | Используется для определения аварийных состояний (прогноз и фактический исход) |
| `forecast_log` | Вставка записи прогноза и факта |
| `markov_frequencies` | Обновляется через `log_transition_and_update` (увеличение частоты) |
| `transition_log` | Вставка записи перехода |

## Зависимости от других функций

| Функция | Назначение | Вызывается в шаге |
|---------|------------|-------------------|
| `apply_forgetting(alpha)` | Экспоненциальное затухание частот | 1 |
| `update_markov_probabilities()` | Пересчёт вероятностей из частот | внутри `apply_forgetting` |
| `get_current_os_waiting_correlation_for_markov_chain()` | Получение метрик | 2 |
| `get_state_id()` | Кодирование трёх параметров в `SMALLINT` | 5 |
| `log_transition_and_update(...)` | Логирование перехода + обновление частот | 9 |

**Косвенно** (через `apply_forgetting` и `update_markov_probabilities`) также используются:
- `rebuild_markov_absorbing()` – перестроение поглощающей матрицы.

## Особенности реализации

### Детерминированность времени вызова
Функция не принимает параметров и всегда работает с текущим временем `now()`. Это делает её удобной для запуска по расписанию (например, `* * * * *` в cron) – каждый вызов обрабатывает очередную минуту.

### Отсутствие блокировок
Все операции используют `INSERT ... ON CONFLICT DO UPDATE` и `UPDATE` без длительных блокировок. Таблицы `markov_chain` содержит одну строку, но обновляется атомарно.

### Устойчивость к пропуску вызовов
Если планировщик пропустил несколько минут, функция всё равно корректно отработает:
- В `markov_chain` останется старые `prev_*` и `curr_*`.
- При следующем вызове сдвиг состояния произойдёт только один раз (пропущенные минуты не будут восстановлены).  
  Это допустимо, так как модель предполагает регулярные вызовы. При редких сбоях лучше восстанавливать состояние из `transition_log`, но текущая архитектура этого не делает – предполагается высокая надёжность планировщика.

### Использование `RAISE NOTICE`
Информационные сообщения помогают отлаживать:
- Пропуск из-за нерабочего времени.
- Значения метрик (включено по умолчанию: `RAISE NOTICE 'new_values_rec=%', new_values_rec;`).

В промышленной среде можно закомментировать или снизить уровень логирования.

## Конфигурация забывания (таблица `markov_config`)

Перед первым запуском необходимо убедиться, что в таблице есть строка:

```sql
INSERT INTO markov_config (last_forget_time, alpha, interval_hours)
VALUES (now(), 0.01, 1)
ON CONFLICT DO NOTHING;
```

Параметры можно менять на лету:

```sql
UPDATE markov_config SET alpha = 0.005, interval_hours = 2;
```

Изменения вступят в силу при следующем вызове `markov_chain_training()` (без перезапуска).

---

## Обработка ошибок и исключительные ситуации

| Ситуация | Поведение |
|----------|-----------|
| Нет данных в `cluster_stat_median` | Функция `get_current_os_waiting_correlation` вернёт нулевые корреляцию и тренды. Обучение продолжится, но переходы могут быть ошибочными. Рекомендуется проверять наличие данных перед вызовом. |
| `markov_chain` пуста | Инициализация (шаг 3). |
| В `markov_probabilities` нет записей для `prev_state` | `risk_pred = 0.0` (прогноз нулевого риска). |
| Деление на ноль при расчёте трендов | Функция `get_current_os_waiting_correlation` перехватывает `division_by_zero` и возвращает тренд 0. |
| Сбой обновления частот (редко, конфликт уникальности) | `ON CONFLICT DO UPDATE` решает проблему. |
| Выход за пределы `SMALLINT` (невозможно, так как состояний 189) | Не возникает. |

```
[cluster_stat_median]  →  get_current_...()  →  markov_chain_training
                                                       ↓
                                    [markov_chain] (состояние)
                                                       ↓
                                    +-----------------+-----------------+
                                    ↓                 ↓                 ↓
                         [forecast_log]     [transition_log]   [markov_frequencies]
                                    ↓                 ↓                 ↓
                              (оценка точности)   (аудит)         (забывание → вероятности)
```

## Рекомендации по мониторингу

1. **Проверять, что функция действительно вызывается каждую минуту в рабочие часы.**  
   Можно подсчитать количество записей в `transition_log` за день и сравнить с ожидаемым (часы работы × 60).

2. **Контролировать `forecast_log`.**  
   Должен содержать столько же записей, сколько вызовов функции (кроме первого). Если записей меньше – возможны ошибки в логировании.

3. **Отслеживать `last_forget_time` в `markov_config`.**  
   Значение должно обновляться каждые `interval_hours` (в идеале – ровно в час). Если не обновляется – забывание не срабатывает.

4. **Анализировать `RAISE NOTICE` сообщения при отладке.**  
   Включите логирование на уровне `INFO` для сеансов планировщика.

5. **Периодически проверять достаточность обучения** через `evaluate_training_sufficiency`.  
   Если критерии не выполняются – корректировать параметры забывания или увеличивать период сбора данных.

## Ограничения и предположения

- **Предполагается, что `cluster_stat_median` обновляется каждую минуту** (или чаще). Если данные приходят реже, модель будет работать с устаревшими метриками.
- **Функция не обрабатывает переход через полночь:** сдвиг состояния происходит исправно, но если вызовы пропущены, данные могут быть не согласованы.
- **Нет механизма восстановления после длительного простоя.** При остановке системы на несколько часов цепь Маркова продолжит обучение с того же состояния, что может привести к артефактам. Рекомендуется при запуске после простоя выполнить принудительное обновление метрик.
- **Состояния кодируются с точностью до 0.1 корреляции.** Быстрые флуктуации внутри этого шага игнорируются – это особенность модели, а не ошибка.

---

## Расширение функциональности (возможные модификации)

При необходимости можно адаптировать функцию:

- **Изменить рабочие часы** – заменить условия на `EXTRACT(HOUR FROM now()) BETWEEN 9 AND 18` и т.п.
- **Добавить гибкий календарь праздников** – через отдельную таблицу `holidays`.
- **Реализовать восстановление после пропусков** – читать последнюю запись из `transition_log`, а не полагаться на `markov_chain`.
- **Увеличить шаг прогноза** – использовать `predict_risk_k_diag` внутри, но это потребует изменения сигнатуры.
- **Параметризовать дату модели** – передавать `model_train_date` как аргумент, чтобы логировать прогнозы для разных версий (сейчас жёстко `current_date`).

---

## Заключение

Функция `markov_chain_training` является **центральным звеном** всей системы марковского анализа производительности PostgreSQL. Она интегрирует сбор метрик, адаптивное забывание, прогнозирование риска и постоянное обновление модели. Благодаря её регулярному вызову (ежеминутно в рабочее время) модель остаётся актуальной и способной обнаруживать дрейф характеристик системы. Логирование прогнозов и фактических исходов позволяет объективно оценивать качество модели и автоматически принимать решение о достаточности обучения.
