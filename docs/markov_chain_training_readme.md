## Инструкция по подготовке запуска и использованию функции `evaluate_training_sufficiency`

### 1. Назначение функции

Функция проверяет **достаточность обучения** цепи Маркова, используемой для прогнозирования риска аварийных состояний. Оценка выполняется по четырём критериям:

| Критерий | Описание | Условие прохождения |
|----------|----------|----------------------|
| **C1** | Для состояний, встречающихся с частотой >1% в `transition_log`, количество переходов `n_i` должно быть не менее 50. | `n_i >= 50` для всех таких состояний |
| **C2** | Максимальное абсолютное изменение вероятностей переходов между текущей матрицей (`markov_probabilities`) и снимком недельной давности (`markov_probabilities_prev_week`) меньше 0.05. | `max|P_new - P_old| < 0.05` |
| **C3** | Улучшение качества прогнозов (Brier Score) при добавлении новой недели обучения не превышает 0.01. Сравниваются две модели (старая и новая) на одном тестовом периоде. | `BS_old - BS_new < 0.01` |
| **C4** | KL-дивергенция между стационарным распределением цепи и эмпирическим распределением состояний за последнюю неделю меньше 0.1. | `KL < 0.1` |

Функция возвращает таблицу с результатами каждого критерия, включая:
- `criterion` – название критерия
- `value` – вычисленное значение (или -1, если данные отсутствуют)
- `threshold` – пороговое значение
- `passed` – логический результат
- `details` – пояснение

### 2. Подготовка данных перед вызовом

#### 2.1. Обязательные предварительные действия

- **Таблица `transition_log`** – должна содержать историю переходов между состояниями. Она заполняется автоматически при вызове `log_transition_and_update()` в процессе обучения.
- **Таблица `markov_probabilities`** – актуальная матрица вероятностей, обновляется через `update_markov_probabilities()`.
- **Таблица `markov_probabilities_prev_week`** – еженедельный снимок матрицы. Создаётся вызовом `snapshot_markov_prev_week()`.
- **Таблица `forecast_log`** – необходима только для критерия C3. Заполняется прогнозами для разных версий моделей (см. раздел 2.3).

#### 2.2. Настройка автоматического создания снимков (для C2 и C4)

Критерии C2 и C4 требуют наличия снимка матрицы за прошлую неделю и данных за последние 7 дней.

**Рекомендуемая конфигурация cron (или pg_cron):**
```sql
-- Каждую пятницу в 18:05 (после окончания рабочей недели)
SELECT cron.schedule('snapshot-markov', '5 18 * * 5', 'SELECT snapshot_markov_prev_week();');
```
Это создаст/обновит таблицу `markov_probabilities_prev_week` и заархивирует текущую матрицу в `markov_probabilities_archive` с датой `current_date`.

#### 2.3. Подготовка данных для критерия C3 (сравнение Brier Score)

Критерий C3 требует, чтобы в таблице `forecast_log` были записи прогнозов **для двух версий модели** (старой и новой) на **одном и том же тестовом периоде** (обычно 1–2 рабочих дня).

##### Шаги подготовки:

1. **Создание снимков моделей**  
   После каждой недели обучения (например, в пятницу) выполните:
   ```sql
   SELECT archive_markov_probabilities(current_date);
   ```
   Это сохранит текущую матрицу в `markov_probabilities_archive` с пометкой `train_date`.

2. **Прогон прогнозов на тестовом периоде**  
   Выберите тестовый период из **дней, не использованных при обучении** проверяемых моделей.  
   Например, если старая модель обучена до `2025-05-12`, а новая – до `2025-05-19`, тестовым периодом могут быть `2025-05-13` и `2025-05-14` (два дня после старой модели, но до обучения новой).

   Для каждой минуты тестового периода выполните прогноз с использованием **фиксированной архивной матрицы**:
   ```sql
   -- Для старой модели (обучена до 2025-05-12)
   SELECT log_forecast(
       p_predicted_risk   := predict_risk_1min_archived('2025-05-12', <from_state>),
       p_actual_risk      := <0 или 1>,
       p_model_train_date := '2025-05-12',
       p_from_state       := <prev_state>,
       p_to_state         := <curr_state>
   );

   -- Для новой модели (обучена до 2025-05-19)
   SELECT log_forecast(
       p_predicted_risk   := predict_risk_1min_archived('2025-05-19', <from_state>),
       p_actual_risk      := <0 или 1>,
       p_model_train_date := '2025-05-19',
       p_from_state       := <prev_state>,
       p_to_state         := <curr_state>
   );
   ```

   **Примечание:** Для автоматизации можно создать процедуру, которая за определённый период:
   - извлекает последовательность состояний из `transition_log`
   - для каждого перехода вычисляет `predicted_risk` по архивной матрице
   - вызывает `log_forecast`.

3. **Проверка наличия данных**  
   Убедитесь, что для обеих дат моделей в `forecast_log` есть записи за тестовый период:
   ```sql
   SELECT model_train_date, COUNT(*)
   FROM forecast_log
   WHERE ts BETWEEN '2025-05-13' AND '2025-05-15'  -- тестовые дни
   GROUP BY model_train_date;
   ```

### 3. Вычисление входных параметров

Функция вызывается с четырьмя параметрами (все могут быть NULL):

```sql
evaluate_training_sufficiency(
    test_start      DATE,   -- начало тестового периода (включительно)
    test_end        DATE,   -- конец тестового периода (включительно)
    model_date_old  DATE,   -- дата обучения старой модели (например, прошлая пятница)
    model_date_new  DATE    -- дата обучения новой модели (например, текущая пятница)
)
```

#### 3.1. Параметры для критерия C3 (обязательны вместе)

- **`test_start`, `test_end`** – границы тестового периода, на котором сравниваются две модели.  
  *Рекомендации:*  
  - Длительность 1–2 полных рабочих дня (например, среда–четверг текущей недели).  
  - Период **не должен использоваться** при обучении ни одной из сравниваемых моделей.  
  - Убедитесь, что за эти дни в `forecast_log` есть записи для обеих `model_train_date`.

- **`model_date_old`** – дата обучения старой модели (обычно дата последнего еженедельного архива до тестового периода).  
- **`model_date_new`** – дата обучения новой модели (обычно дата архива, созданного после тестового периода).

**Пример выбора дат:**  
- Старая модель обучена до 2025-05-12 (пятница).  
- Тестовый период: 2025-05-13 и 2025-05-14 (вторник–среда).  
- Новая модель обучена до 2025-05-19 (пятница).  
Тогда вызов:  
`test_start = '2025-05-13', test_end = '2025-05-14', model_date_old = '2025-05-12', model_date_new = '2025-05-19'`

#### 3.2. Если параметры не заданы (все NULL)

Критерий C3 будет пропущен с сообщением:  
`'Не заданы параметры тестового периода/моделей. Выполните еженедельное сравнение вручную.'`  
Остальные критерии (C1, C2, C4) будут вычислены на основе текущих данных.

### 4. Запуск функции и интерпретация результатов

#### 4.1. Типовой вызов (все критерии)

```sql
SELECT * FROM evaluate_training_sufficiency(
    test_start      => '2025-05-13',
    test_end        => '2025-05-14',
    model_date_old  => '2025-05-12',
    model_date_new  => '2025-05-19'
);
```

#### 4.2. Пример результата

| criterion                               | value | threshold | passed | details                                           |
|-----------------------------------------|-------|-----------|--------|---------------------------------------------------|
| C1: n_i >= 50 (для частых >1%)          | 0     | 0         | true   | Состояний с частотой >1%: 23, из них с n_i<50: 0 |
| C2: max \|P_new - P_old\|               | 0.034 | D < 0.05  | true   | D_max = 0.0340                                    |
| C3: Brier Score изменение < 0.01        | 0.007 | < 0.01    | true   | Изменение BS = 0.0070                             |
| C4: KL(pi \|\| emp) < 0.1               | 0.082 | < 0.1     | true   | KL = 0.0820                                       |

Если все `passed = true` – обучение достаточно.  
Если какой-либо критерий не выполнен, следуйте рекомендациям из раздела 5.

#### 4.3. Вызов без параметров (только C1, C2, C4)

```sql
SELECT * FROM evaluate_training_sufficiency();
```

### 5. Рекомендации при невыполнении критериев

| Критерий | Возможная причина | Действие |
|----------|------------------|----------|
| **C1**   | Недостаточно данных для часто встречающихся состояний. | Продолжить сбор данных, увеличить период обучения. |
| **C2**   | Вероятности переходов продолжают существенно меняться. | Увеличить интервал забывания (alpha) или добавить больше данных. |
| **C3**   | Качество прогнозов улучшается более чем на 0.01 Brier Score. | Добавить ещё одну неделю обучения и повторить тест. |
| **C4**   | Стационарное распределение не соответствует недельной эмпирике. | Профиль нагрузки изменился – уменьшить период забывания или обучить отдельные матрицы для разных часов/дней недели. |

### 6. Автоматизация еженедельной проверки

Настройте задание cron (внешний планировщик или pg_cron) следующего содержания:

1. **Пятница, 18:05** – `SELECT snapshot_markov_prev_week();` (уже описано).
2. **Суббота, 00:30** – прогон прогнозов для старой и новой модели на тестовом периоде (например, среда–четверг прошедшей недели) с заполнением `forecast_log`.
3. **Суббота, 01:00** – вызов `evaluate_training_sufficiency` с соответствующими датами и запись результата в таблицу `training_sufficiency_log`.

Пример логирования результата:
```sql
CREATE TABLE training_sufficiency_log (
    check_date TIMESTAMPTZ DEFAULT now(),
    test_start DATE,
    test_end DATE,
    model_date_old DATE,
    model_date_new DATE,
    all_passed BOOLEAN,
    details JSONB
);

-- После вызова функции можно вставить результат:
INSERT INTO training_sufficiency_log (test_start, test_end, model_date_old, model_date_new, all_passed, details)
SELECT
    '2025-05-13', '2025-05-14', '2025-05-12', '2025-05-19',
    bool_and(passed),
    jsonb_agg(jsonb_build_object('criterion', criterion, 'value', value, 'passed', passed))
FROM evaluate_training_sufficiency('2025-05-13', '2025-05-14', '2025-05-12', '2025-05-19');
```

### 7. Важные замечания

- Функция **не изменяет данные**, только анализирует существующие таблицы.
- Критерий C2 требует наличия таблицы `markov_probabilities_prev_week`. Если она пуста или отсутствует, `passed = false` с пояснением.
- Критерий C4 использует `transition_log` за **последние 7 дней**. Если записей нет, `passed = false`.
- Для корректной работы C3 необходимо, чтобы в `forecast_log` были записи с `model_train_date`, равными `model_date_old` и `model_date_new`, за интервал `[test_start, test_end]`. Иначе `value = -1` и `passed = false`.
