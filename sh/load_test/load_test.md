# Нагрузочное тестирование с помощью pgbench (load_test.sh)

**Версия:** 4.0

## Назначение

Скрипт `load_test.sh` предназначен для автоматизированного нагрузочного тестирования PostgreSQL с использованием утилиты `pgbench`. Он управляет циклами тестирования, сбором метрик и последовательным выполнением сценариев нагрузки. Скрипт интегрируется с управляющей базой данных (`expecto_db`), где хранятся настройки тестов и собираются результаты.

## Основные возможности

- Пошаговое выполнение нагрузки с заданным количеством клиентов (`pgbench_clients`).
- Поддержка нескольких тестовых сценариев (SQL-файлы `do_scenario*.sql`).
- Автоматический сбор данных до и после прогона.
- Гибкое переключение между стандартной (`pgbench_db`) и пользовательской тестовой БД.
- Управление через файлы-флаги для безопасного запуска из cron.
- Логирование хода выполнения и ошибок.

## Требования

- **ОС:** Linux (проверено на дистрибутивах с bash).
- **PostgreSQL:** установлен и настроен, доступны БД:
  - `expecto_db` – управляющая БД (содержит метаданные теста).
  - `pgbench_db` – стандартная тестовая БД (или пользовательская).
- **Пользователь:** `expecto_user` с правами на обе БД.
- **Утилиты:** `pgbench`, `psql`, `grep`, `readlink`, `dirname`, `touch`, `rm`.
- **Дополнительные скрипты:** 
  - `get_conf_param.sh` – получение параметров конфигурации (например, имя тестовой БД).
  - `load_test_stop.sh` – остановка теста.
  - Файлы сценариев `do_scenario1.sql`, `do_scenario2.sql`, ... – запросы для pgbench.
- **Файлы-флаги** (создаются автоматически):
  - `LOAD_TEST_STARTED` – сигнал о необходимости запуска теста.
  - `LOAD_TEST_IN_PROGRESS` – блокировка повторного запуска.
  - `PGBENCH_WORKING` – индикатор активной нагрузки.

## Установка и настройка

1. **Подготовьте тестовые сценарии**:
   - Создайте файлы `do_scenario1.sql`, `do_scenario2.sql`, ... в той же директории. Каждый файл содержит SQL-запросы, которые будут выполняться pgbench во время соответствующего сценария.

2. **Настройте параметры**:
   - В файле `get_conf_param.sh` реализуйте получение параметра `testdb`. Он может возвращать:
     - `default` – использовать `pgbench_db` как тестовую БД.
     - Имя другой БД – для кастомной нагрузки.
   - При необходимости измените переменную `pg_bench_time` (по умолчанию 600 секунд) внутри скрипта.

3. **Добавьте задание в crontab** :
   ```
   */1 * * * * /postgres/pg_expecto/load_test/load_test.sh
   ```

## Использование

### Запуск теста


### Остановка теста

- Для принудительной остановки (например, по расписанию или при ошибке) используется скрипт `load_test_stop.sh`. Он должен удалять флаг `LOAD_TEST_STARTED` и выполнять другие необходимые действия (очистка, завершение процессов).
- Если тест был остановлен принудительно, таблица `pgbench_history` (в стандартной БД) будет очищена, и формирование отчёта не выполнится.

### Мониторинг

- **Лог выполнения:** `load_test.log` – содержит временные метки, информацию о текущем проходе, количестве сессий pgbench, статусе сценариев.
- **Лог ошибок:** `load_test.err` – записываются сообщения об ошибках, возвращаемые psql и pgbench.
- **Файл прогресса:** `load_test.progress` – используется для перенаправления stderr pgbench (содержит детали выполнения команд).
- **Файл временных меток:** `timestamp.log` – не используется в скрипте напрямую, но может быть задействован в других компонентах.

```

├── load_test_stop.sh          # Скрипт остановки теста
├── get_conf_param.sh          # Получение конфигурации (testdb)
├── do_scenario1.sql           # Сценарий 1
├── do_scenario2.sql           # Сценарий 2
├── ...                        # Другие сценарии

├── PGBENCH_WORKING            # Флаг выполнения pgbench (авто)
├── load_test.log              # Лог выполнения
├── load_test.err              # Лог ошибок
├── load_test.progress         # Прогресс pgbench
└── timestamp.log              # Дополнительный лог (по желанию)
```

## Детали работы

### Основной цикл

1. Проверка наличия флага `LOAD_TEST_STARTED` – если нет, выход.
2. Проверка флага `LOAD_TEST_IN_PROGRESS`:
   - Если есть – значит, тест уже запущен, и это повторный вызов. В таком случае:
     - Получается текущий номер прохода (`current_pass`) и количество клиентов.
     - Вызывается функция `load_test_set_scenario_queryid()` для фиксации queryid сценария.
     - Скрипт завершается.
   - Если нет – запускается новая итерация.

3. **Новая итерация:**
   - Создаётся `LOAD_TEST_IN_PROGRESS`.
   - Вызывается `load_test_start_collect_data()` – сбор начальных метрик.
   - Определяется `current_pass`, `pgbench_clients` (количество клиентов для текущего прохода).
   - Устанавливается время прогона `pg_bench_time=600`.
   - Вычисляется число потоков (`jobs`) по количеству процессоров.
   - Получается имя тестовой БД через `get_conf_param.sh` (переменная `testdb`).
   - Получается идентификатор текущего теста (`current_test_id`) и количество сценариев (`max_sc_count`).
   - В зависимости от `testdb`:
     - **Если `default`** – используется БД `pgbench_db`. Для каждого сценария:
       - Получается количество клиентов для сценария.
       - Запускается `pgbench` в фоне с параметрами:
         ```
         pgbench --username=expecto_user --file=do_scenarioN.sql --protocol=extended --report-per-command --jobs=<jobs> --client=<clients> --time=600 pgbench_db
         ```
       - Все фоновые процессы ждутся через `wait`.
     - **Если не `default`** – используется кастомная БД. Дополнительно определяется владелец БД, и pgbench запускается с параметром `--no-vacuum` и с указанием владельца как имени пользователя.
   - После завершения pgbench удаляется флаг `PGBENCH_WORKING`.

4. Остановка сбора данных: `load_test_stop_collect_data()`.

5. Проверка принудительной остановки:
   - Если флаг `LOAD_TEST_STARTED` был удалён (внешним воздействием), очищается таблица `pgbench_history` (только для `default` БД) и скрипт завершается.

6. Проверка условия завершения теста:
   - Вызывается `load_test_is_test_could_be_finished()`. Если возвращает `1`:
     - Для `default` БД очищается `pgbench_history`.
     - Выполняется `load_test_stop.sh` (удаляет флаги).
     - Логируется завершение.
   - Если тест не завершён, удаляется флаг `LOAD_TEST_IN_PROGRESS`.

### Обработка ошибок

- Функция `exit_code` проверяет код возврата последней команды. При ненулевом коде:
  - Записывается сообщение об ошибке в лог и в `load_test.err`.
  - Вызывается `load_test_stop.sh` для корректной остановки теста.
  - Скрипт завершается с соответствующим кодом.

## Параметры конфигурации

Параметры, которые могут быть изменены в скрипте:

| Переменная        | Описание                                                                                   | Значение по умолчанию |
|-------------------|--------------------------------------------------------------------------------------------|------------------------|
| `pg_bench_time`   | Длительность прогона pgbench (секунды)                                                      | 600                    |
| `expecto_db`      | Имя управляющей БД                                                                         | `expecto_db`           |
| `pgbench_db`      | Имя стандартной тестовой БД (используется при `testdb=default`)                            | `pgbench_db`           |
| `expecto_user`    | Пользователь PostgreSQL для подключения                                                     | `expecto_user`         |

Остальные настройки (количество клиентов, сценарии, условие завершения) задаются через функции в управляющей БД.

## Возможные проблемы и их решение

1. **Скрипт не запускается:**
   - Проверьте права на выполнение (`chmod +x load_test.sh`).
   - Убедитесь, что в директории есть файл `LOAD_TEST_STARTED`.
   - Посмотрите в `load_test.err` на наличие ошибок подключения к БД.

2. **Ошибка при запуске pgbench:**
   - Проверьте, что пользователь `expecto_user` имеет доступ к тестовой БД и права на выполнение сценариев.
   - Убедитесь, что файлы `do_scenario*.sql` существуют и корректны.

3. **Тест не завершается:**
   - Убедитесь, что функция `load_test_is_test_could_be_finished()` возвращает `1` при достижении нужного количества проходов.
   - Проверьте наличие флага `LOAD_TEST_STARTED` – возможно, он не был удалён после завершения.

4. **Нет сбора данных:**
   - Убедитесь, что функции `load_test_start_collect_data()` и `load_test_stop_collect_data()` реализованы в БД и не возвращают ошибок.

5. **Множественные параллельные запуски:**
   - Флаг `LOAD_TEST_IN_PROGRESS` предотвращает повторный запуск до окончания текущей итерации. Если скрипт завис, удалите этот флаг вручную и перезапустите.

## Примечания по безопасности

- Не рекомендуется запускать скрипт с правами суперпользователя PostgreSQL. Используйте выделенного пользователя `expecto_user` с минимальными необходимыми привилегиями.
- Убедитесь, что директория со скриптами защищена от несанкционированного доступа, так как она содержит SQL-файлы и логи.
