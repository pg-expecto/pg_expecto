### Подробное описание сервисного скрипта `run_incident.sh`

#### 1. Назначение
Скрипт `run_incident.sh` предназначен для имитации инцидентов производительности СУБД PostgreSQL путём выполнения операции `VACUUM FREEZE` с управляемыми параметрами задержки и лимита стоимости (`vacuum_cost_delay`, `vacuum_cost_limit`). Это создаёт дополнительную нагрузку на базу данных и позволяет тестировать реакцию системы мониторинга, а также проверять устойчивость приложения в условиях замедления работы СУБД. Скрипт может работать как по заранее сгенерированному расписанию, так и по требованию, при обнаружении недостаточного количества инцидентов за последние 24 часа.

#### 2. Место в общем процессе нагрузочного тестирования
В рамках нагрузочного тестирования скрипт запускается из основного скрипта `load_test.sh` (при установленном параметре `vacuum_incident=1`) в каждой итерации теста. Вызов происходит асинхронно в фоновом режиме с перенаправлением вывода в лог-файл:
```bash
/postgres/pg_expecto/sh/load_test/run_incident.sh >> /postgres/pg_expecto/sh/load_test/run_incident.log 2>&1
```
Это позволяет имитировать инцидент параллельно с основной нагрузкой `pgbench`, не блокируя выполнение самого теста.

#### 3. Конфигурируемые параметры (переменные окружения)
Скрипт использует следующие переменные окружения (со значениями по умолчанию):

| Переменная | Значение по умолчанию | Назначение |
|------------|----------------------|------------|
| `PGDATABASE` | `expecto_db` | Имя базы данных для подключения (управляющей БД, где хранится расписание и история) |
| `PGUSER` | `expecto_user` | Пользователь для подключения |
| `PGHOST` | `localhost` | Хост СУБД |
| `PGPORT` | `5432` | Порт |
| `VACUUM_COST_DELAY` | `20` | Задержка (мс) для `vacuum_cost_delay` |
| `VACUUM_COST_LIMIT` | `200` | Лимит стоимости для `vacuum_cost_limit` |
| `TABLE_NAME` | `pgbench_accounts` | Таблица, на которой выполняется `VACUUM FREEZE` |
| `LOCK_DIR` | `/tmp` | Каталог для файла блокировки |
| `MIN_INCIDENT_DURATION_SECONDS` | `30` | Минимальная длительность выполнения `VACUUM`, чтобы засчитать инцидент (сек) |
| `MAX_RETRIES` | `5` | Максимальное число повторных попыток для события, если его длительность оказалась меньше порога |
| `TARGET_INCIDENTS_PER_DAY` | `2` | Целевое количество инцидентов за последние 24 часа |
| `PSQL` | `psql` | Команда для вызова `psql` |

#### 4. Структура и логика работы

##### 4.1. Генерация расписания
При запуске скрипт вызывает функцию базы данных `generate_incident_schedule(7)`, которая создаёт записи в таблице `incident_schedule` на ближайшие 7 дней (вероятно, с некоторой периодичностью). Это делается для того, чтобы всегда иметь запланированные события.

##### 4.2. Блокировка (flock)
Используется блокировка через `flock` на файл `$VACUUM_LOCK_FILE` (по умолчанию `/tmp/incident_vacuum.lock`), чтобы предотвратить параллельный запуск нескольких экземпляров скрипта. Если блокировку не удаётся захватить, скрипт завершается, выводя сообщение.

##### 4.3. Проверка количества инцидентов за 24 часа
Выполняется SQL-запрос к функции `count_incidents_last_24h()`, которая возвращает число инцидентов, зафиксированных в таблице `performance_incident` за последние сутки. Если это число меньше целевого (`TARGET_INCIDENTS_PER_DAY`), скрипт создаёт **внеплановое** событие, вставляя запись в `incident_schedule` с `scheduled_start = now()`, длительностью 5 минут, приоритетом 4 и нулевым числом попыток. Затем сразу обрабатывает это событие функцией `process_event`.

##### 4.4. Поиск запланированного события
Выполняется запрос:
```sql
SELECT id FROM incident_schedule
WHERE executed = FALSE AND scheduled_start <= now()
ORDER BY scheduled_start LIMIT 1
```
Если найдено событие, и его ID не совпадает с только что созданным внеплановым (чтобы не обрабатывать дважды), вызывается `process_event`.

##### 4.5. Обработка события (`process_event`)
Функция принимает ID события. Она:
- Получает запланированное время и приоритет из таблицы.
- Выполняет `VACUUM FREEZE` на целевой таблице с установленными `vacuum_cost_delay` и `vacuum_cost_limit`.
- Замеряет время выполнения.
- **Если длительность ≥ `MIN_INCIDENT_DURATION_SECONDS`**:  
  Вставляется запись в таблицу `performance_incident` с указанием приоритета и временных меток. Обновляется запись расписания: устанавливается `executed = TRUE`, фиксируются фактические время начала и окончания.
- **Если длительность < порога**:  
  Инкрементируется `retry_count`. Если достигнут `MAX_RETRIES`, событие помечается выполненным (но без инцидента). Иначе запись обновляется с новым числом попыток, и событие остаётся в очереди для повторной попытки при следующем запуске.

Во всех случаях возвращается код:
- `0` – инцидент зафиксирован;
- `1` – лимит попыток исчерпан;
- `2` – попытка не удалась, будет повтор.

##### 4.6. Завершение
Скрипт завершается после обработки внепланового и/или запланированного события (или при их отсутствии). Процесс блокировки освобождается по выходу из подоболочки.

#### 5. Взаимодействие с базой данных
- Все SQL-запросы выполняются через `psql` с параметрами подключения.
- Используются функции:
  - `generate_incident_schedule(7)` – генерация расписания.
  - `count_incidents_last_24h()` – подсчёт инцидентов.
  - `process_event` использует прямые `UPDATE`/`INSERT`.
- Все обращения к БД выполняются с опцией `-t -A` для скалярных значений, а для `VACUUM` используется отдельный вызов с включённым `ON_ERROR_STOP=1`.

#### 6. Обработка ошибок
В скрипте установлена опция `set -euo pipefail`, что обеспечивает остановку при любой ошибке (кроме явно обработанных), а также защиту от неопределённых переменных. Ошибки выполнения SQL-команд (в `process_event`) могут приводить к аварийному завершению, но это не критично для общего процесса, так как скрипт запускается в фоне и его сбой не останавливает нагрузочный тест.

#### 7. Логирование
Все сообщения (`echo`) выводятся в стандартный вывод, который при вызове из `load_test.sh` перенаправляется в файл `run_incident.log`. Сообщения содержат временные метки, ID событий, длительность выполнения и статус обработки.

#### 8. Влияние на нагрузку
- `VACUUM FREEZE` с заниженным `vacuum_cost_delay` и увеличенным `vacuum_cost_limit` создаёт интенсивную I/O-нагрузку и блокировки, имитируя проблему производительности.
- Скрипт может запускаться несколько раз в течение одного теста (каждую минуту по расписанию `cron`), но благодаря блокировке одновременно выполняется только один экземпляр.
- Если событие не достигло порога длительности, оно повторяется (до 5 раз), что увеличивает шанс создания заметного инцидента.

#### 9. Интеграция с таблицами
Предполагаемая структура таблиц:
- `incident_schedule` – хранит запланированные события: `id`, `scheduled_start`, `duration_minutes`, `priority`, `executed` (bool), `actual_start`, `actual_finish`, `retry_count`.
- `performance_incident` – хранит зафиксированные инциденты: `id`, `priority`, `start_timepoint`, `finish_timepoint`.

#### 10. Рекомендации по использованию
- Перед применением необходимо создать вышеуказанные таблицы и функции в управляющей БД.
- Переменные окружения могут быть заданы в сеансе или в оболочке запуска, например, в `cron`.
- Целесообразно запускать скрипт периодически (например, каждые 5 минут) через `cron`, чтобы обеспечить достаточное количество событий в течение суток.
- Значения `VACUUM_COST_DELAY` и `VACUUM_COST_LIMIT` следует подбирать эмпирически, чтобы длительность `VACUUM` превышала порог `MIN_INCIDENT_DURATION_SECONDS`.

#### 11. Пример вызова
```bash
export PGDATABASE=expecto_db PGUSER=expecto_user
export VACUUM_COST_DELAY=10 VACUUM_COST_LIMIT=500
./run_incident.sh
```

#### 12. Особенности реализации
- Использование `exec_sql_scalar` для получения скалярных значений с подавлением ошибок (`2>/dev/null`), что предотвращает преждевременный выход при отсутствии данных.
- Блокировка через `flock` обеспечивает безопасность при параллельных запусках.
- Возможность ручной генерации внепланового события при нехватке инцидентов – это позволяет поддерживать заданное количество инцидентов даже при низкой эффективности запланированных событий.

Данный скрипт является ключевым элементом системы нагрузочного тестирования для проверки механизмов обнаружения и реагирования на проблемы производительности PostgreSQL.
