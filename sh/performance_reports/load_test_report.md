# load_test_report.sh

**Версия:** 7.4  
**Дата последнего обновления:** 26.03.2026  

## Обзор

`load_test_report.sh` — это shell-скрипт для формирования отчётов по результатам нагрузочного тестирования PostgreSQL. Он входит в состав инструментария **pg_expecto** и предназначен для сбора и анализа метрик производительности СУБД, событий ожидания и системных показателей во время проведения тестов.

Скрипт получает из базы данных информацию о текущем прогоне, генерирует различные отчёты (статистику, корреляционный анализ, диаграммы Парето) и создаёт структурированные файлы с промптами для последующего анализа с помощью больших языковых моделей (например, ChatGPT). Все сформированные файлы сохраняются в каталог `/tmp/pg_expecto_reports/`.

## Требования

- **Операционная система:** Linux (используются `bash`, `psql`, `lscpu`, `lsblk`, `free`, `readlink`, `dirname` и др.)
- **PostgreSQL:** Запущенный экземпляр PostgreSQL с базой данных `expecto_db` и ролью `expecto_user`.
- **Расширения PostgreSQL:** В базе данных должны присутствовать функции, предоставляемые pg_expecto:
  - `load_test_get_current_test_id()`
  - `report_load_test_loading()`
  - `report_queryid_stat(queryid, wait_event_type, start_timestamp, finish_timestamp)`
  - `get_vm_params_list()`
- **Интерпретатор:** `bash` (скрипт использует `[[ ]]`, что является расширением bash).
- **Права доступа:** Пользователь, запускающий скрипт, должен иметь возможность:
  - Подключаться к `expecto_db` от имени `expecto_user` (предполагается, что аутентификация настроена через `.pgpass` или доверительное соединение).
  - Записывать файлы в `/tmp/pg_expecto_reports/`.
  - Выполнять сопутствующие скрипты `summary_report.sh` и `get_reports_param.sh`, которые должны находиться в том же каталоге.

## Использование

Запустите скрипт из командной строки:

```bash
./load_test_report.sh
```

Скрипт не принимает аргументов командной строки — все настройки заданы внутри.

### Пример

```bash
cd /path/to/pg_expecto_scripts
./load_test_report.sh
```

После завершения работы все отчёты будут находиться в каталоге `/tmp/pg_expecto_reports/`. Для диагностики можно просмотреть файлы журнала `load_test_report.log` и ошибок `load_test_report.err` в каталоге скрипта.

## Рабочий процесс

1. **Инициализация**
   - Определяются пути к файлам журнала и ошибок.
   - Удаляются все файлы из каталога `/tmp/pg_expecto_reports/` (при необходимости).

2. **Получение контекста теста**
   - Запрос текущего идентификатора теста через функцию `load_test_get_current_test_id()`.
   - Получение временных меток начала и окончания теста.

3. **Отчёт по изменению нагрузки**
   - Выполнение `report_load_test_loading()` для генерации графика нагрузки (сохраняется как `x.load_test_loading.txt`).

4. **Сводный отчёт**
   - Вызов `summary_report.sh` с переданными временными метками для формирования сводки по производительности и ожиданиям.

5. **Отчёты по SQL‑сценариям**
   - Для каждого SQL‑сценария (по `queryid`) и для каждого типа событий ожидания (`BufferPin`, `Extension`, `IO`, `IPC`, `Lock`, `LWLock`, `Timeout`) выполняется функция `report_queryid_stat()`.
   - Результаты сохраняются в файлы с именами `scenario.<номер_сценария>.<queryid>.<тип_ожидания>.txt`.

6. **Формирование статистических данных для нейросети (AI)**
   - **Настройки:** Запись настроек PostgreSQL, информации о CPU, размере RAM, блочных устройствах и параметрах VM в файл `_1.settings.txt`.
   - **PostgreSQL и vmstat:** Объединение ранее сгенерированных файлов в `_2.postgresql_vmstat.txt`.
   - **Статистика I/O:** Конкатенация файлов iostat для каждого устройства в `_3.vmstat_iostat.txt`.
   - **Промпты для AI:** Создание текстовых файлов с шаблонами запросов для анализа:
     - `_4.1.performance_prompt.txt` – полный анализ производительности.
     - `_4.2.performance_prompt.diff.txt` – сравнительный анализ двух экспериментов.
     - `_4.3.performance_prompt.short.diff.txt` – краткий сравнительный анализ.
     - `_4.4.performance_prompt.advice.txt` – рекомендации.
     - `_4.5.io_prompt.txt` – анализ подсистемы ввода-вывода.
     - `_4.6.io_prompt.diff.txt` – сравнительный анализ подсистемы ввода-вывода.
     - `_4.7.io_prompt.advice.txt` – рекомендации по I/O.
     - `_5.prompt.advice.majority_vote.txt` – анализ методом большинства голосов.

7. **Завершение**
   - Запись финальных сообщений в журнал и выход с кодом 0.

## Выходные файлы

Все сгенерированные файлы сохраняются в каталоге `/tmp/pg_expecto_reports/`. Основные из них:

| Файл | Описание |
|------|----------|
| `x.load_test_loading.txt` | Данные об изменении нагрузки во времени (график). |
| `scenario.*.txt` | Статистика событий ожидания по каждому SQL‑сценарию. |
| `_1.settings.txt` | Конфигурация СУБД и виртуальной машины. |
| `_2.postgresql_vmstat.txt` | Объединённые данные корреляции PostgreSQL и vmstat. |
| `_3.vmstat_iostat.txt` | Объединённые данные vmstat и iostat для всех устройств. |
| `_4.*.txt` | Промпты для AI‑анализа. |
| `_5.prompt.advice.majority_vote.txt` | Промпт для анализа методом большинства голосов. |

Кроме того, в каталоге скрипта создаются:
- `load_test_report.log` – подробный журнал выполнения.
- `load_test_report.err` – вывод ошибок от выполняемых команд.

## Коды возврата и обработка ошибок

- В скрипте определена функция `exit_code`, которая проверяет код возврата последней команды. При ненулевом коде в журнал и файл ошибок записывается сообщение, после чего скрипт завершается с тем же кодом.
- Критическими считаются команды `psql` (при выполнении SQL‑функций) и вызов `summary_report.sh`. В случае их сбоя выполнение прерывается.
- При успешном завершении скрипт возвращает `0`.

## Примечания

- Скрипт написан с использованием русского языка в сообщениях журнала и ошибок.
- Он сильно зависит от функций, предоставляемых расширением pg_expecto. Если эти функции отсутствуют в базе данных, выполнение завершится с ошибкой.
- Сопутствующие скрипты `summary_report.sh` и `get_reports_param.sh` должны находиться в том же каталоге и иметь права на выполнение.
- На первом этапе скрипт удаляет все файлы из каталога `/tmp/pg_expecto_reports/` (команда `rm /tmp/pg_expecto_reports/*`). Будьте осторожны, если этот каталог используется другими процессами.
