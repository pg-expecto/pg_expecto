# Performance Reports – генерация отчётов о производительности PostgreSQL

Данный набор скриптов предназначен для автоматического сбора, анализа и формирования детализированных отчётов о производительности кластера PostgreSQL, системных метриках (vmstat, iostat) и корреляциях между ними. Скрипты используют данные, собранные в базе данных `expecto_db`, и сохраняют результаты в `/tmp/pg_expecto_reports`.

## Обзор скриптов

### `reports.conf`
Файл конфигурации, содержащий настройки для генерации отчётов. В текущей версии задаётся список дисковых устройств для анализа (например, `devices_list = vdd vdc`).

### `get_reports_param.sh`
Вспомогательный скрипт, который читает значение параметра из `reports.conf`. Используется другими отчётами для получения списка устройств.

### `summary_report.sh`
**Ядро системы отчётов.** Генерирует комплексный отчёт за указанный временной интервал. Создаёт следующие файлы в `/tmp/pg_expecto_reports/` (префикс может меняться в зависимости от режима):
- `1.1.postgresql.wait_event_type.txt` – статистический анализ ожиданий СУБД.
- `1.2.vmstat.performance.txt` – трендовый анализ производительности vmstat.
- `1.3.wait_event_type_vmstat.txt` – корреляционный анализ между ожиданиями и метриками vmstat.
- `2.1.vmstat_iostat_<device>.txt` – корреляция vmstat и iostat для каждого устройства.
- `x.postgresql.cluster_performance.txt` – исходные данные производительности кластера.
- `x.vmstat.txt`, `x.iostat_<device>.txt` – данные для построения графиков.

Поддерживает два режима:
- `DEFAULT` (по умолчанию) – обычный отчёт.
- `INCIDENT` – отчёт, помечаемый как относящийся к инциденту (используется префикс `test` в именах файлов).

### `performance_report.sh`
Генерирует отчёт по производительности за произвольный временной интервал, переданный параметрами. Фактически вызывает `summary_report.sh` с переданными интервалами и режимом `DEFAULT`. Дополнительно создаёт:
- `_1.settings.txt` – настройки PostgreSQL и VM.
- `_2.1.postgresql_vmstat.txt` – объединённый корреляционный анализ.
- `_3.1.prompt.txt` – шаблон для аналитической системы (нейросети) с инструкциями по формированию отчёта.

### `incident_report.sh`
Специализированный скрипт для анализа инцидентов. Принимает временную метку окончания инцидента и автоматически вычисляет два интервала:
- **Интервал инцидента:** от `finish_incident - 1 час` до `finish_incident`.
- **Тестовый интервал (сравнение):** от `start_incident - 1 час` до `start_incident`.

Для каждого интервала генерируются отчёты с помощью `summary_report.sh` в режиме `INCIDENT`. Также создаётся общий файл настроек `_1.settings.txt` и сводные отчёты `_2.postgresql_vmstat.txt`, `_2.1.test.postgresql_vmstat.txt`.

### `load_test_report.sh`
Предназначен для анализа нагрузочного тестирования. Автоматически определяет текущий тест (используя функцию `load_test_get_current_test_id()` в базе данных) и временные рамки (минимальный и максимальный временные метки среди прогонов с `pass_counter >= 6`). Далее:
- Генерирует график изменения нагрузки (`x.load_test_loading.txt`).
- Вызывает `summary_report.sh` для основного интервала.
- Для каждого SQL-сценария (из таблицы `testing_scenarios`) создаёт отчёты по каждому типу ожидания (`BufferPin`, `Extension`, `IO`, `IPC`, `Lock`, `LWLock`, `Timeout`) с помощью `report_queryid_stat()`.

### `queryid_report.sh`
Генерирует детальные отчёты по истории выполнения конкретного запроса (по `queryid`) для указанного временного интервала. Для каждого типа ожидания создаётся файл `queryid.<queryid>.<wait_event_type>.txt`, содержащий данные из `reports_queryid_stat()`.

## Конфигурация

Все скрипты используют следующие переменные:
- **Путь к логам и ошибкам:** в текущей директории создаются `.log` и `.err` файлы (название соответствует имени скрипта).
- **Директория для отчётов:** `/tmp/pg_expecto_reports`. При запуске очищается (кроме файлов, используемых внутри процесса).
- **База данных и пользователь:** по умолчанию `expecto_db` и `expecto_user`. Все запросы выполняются через `psql` с этими параметрами.
- **Конфигурация устройств:** читается из `reports.conf` через `get_reports_param.sh`.

## Общая структура отчётов

Все файлы сохраняются в `/tmp/pg_expecto_reports/`. Основные типы файлов:

| Файл | Содержание |
|------|------------|
| `_1.settings.txt` | Версия PostgreSQL, настройки из `pg_settings`, CPU, RAM, список устройств, параметры VM, методология анализа. |
| `_2.postgresql_vmstat.txt` | Сводный корреляционный анализ: ожидания СУБД, производительность vmstat, их корреляция, диаграммы Парето. |
| `1.1.*.wait_event_type.txt` | Статистика по типам ожиданий (wait_event_type) с метриками корреляции, ВКО, R². |
| `1.2.*.vmstat.performance.txt` | Трендовый анализ vmstat-метрик. |
| `1.3.*.wait_event_type_vmstat.txt` | Корреляция между ожиданиями СУБД и метриками vmstat. |
| `1.4.*.wait_event_type_pareto.txt` | Диаграмма Парето для ожиданий. |
| `1.5.*.queryid_pareto.txt` | Диаграмма Парето для запросов (по queryid). |
| `2.1.*.vmstat_iostat_<device>.txt` | Корреляция vmstat и iostat для конкретного устройства. |
| `x.*.txt` | Исходные данные для построения графиков (производительность кластера, vmstat, iostat). |

Имена файлов для тестового интервала (режим `INCIDENT`) содержат дополнительную метку, например `1.1.1.test.postgresql.wait_event_type.txt`.

## Методология анализа

В скриптах реализована трёхэтапная методология статистического анализа ожиданий СУБД и двухэтапная для системных метрик. Критерии оценки подробно описаны в генерируемых отчётах (например, в `_1.settings.txt`):

1. **Статистическая значимость корреляции (p-value)**
   - `p-value < 0.05` – корреляция значима.
   - `p-value ≥ 0.05` – связь нестабильна, интерпретация не рекомендуется.

2. **Взвешенная корреляция ожиданий (ВКО)**
   - `≥ 0.2` – критическое значение, требуется немедленное вмешательство.
   - `[0.1; 0.2)` – высокое значение, требуется глубокий анализ.
   - `[0.04; 0.1)` – среднее значение, контекстный анализ.
   - `[0.01; 0.04)` – низкое значение, наблюдение.
   - `< 0.01` – игнорируется.

3. **Коэффициент детерминации R²**
   - `≥ 0.8` – исключительно сильная модель.
   - `[0.6; 0.8)` – качественная модель.
   - `[0.4; 0.6)` – приемлемая модель.
   - `[0.2; 0.4)` – слабая модель.
   - `< 0.2` – непригодная модель.

Для анализа системных метрик (vmstat, iostat) используется двухэтапная процедура: оценка значимости корреляции (p-value) и интерпретация R².

## Зависимости

- **PostgreSQL** с установленным расширением `pg_expecto` (или пользовательскими функциями, такими как `report_postgresql_wait_event_type`, `get_vm_params_list`, `report_vmstat_performance`, `report_wait_event_type_vmstat`, `report_vmstat_iostat`, `report_postgresql_cluster_performance`, `report_vmstat`, `report_iostat`, `report_load_test_loading`, `reports_queryid_stat`, `load_test_get_current_test_id`, `get_hour_before`). Эти функции не входят в данный набор скриптов.
- Утилита `psql` в PATH.
- Bash-совместимая оболочка (скрипты используют `#!/bin/sh`, но некоторые конструкции специфичны для bash, например `[[`).

## Примеры использования

### 1. Сводный отчёт за произвольный интервал
```bash
./performance_report.sh "2026-03-31 10:00" "2026-03-31 11:00"
```
Создаст отчёты в `/tmp/pg_expecto_reports/` с префиксом `_`, включая настройки и комплексный анализ.

### 2. Анализ инцидента
```bash
./incident_report.sh "2026-03-31 10:00"
```
Автоматически вычислит интервалы инцидента (с 09:00 до 10:00) и сравнительный интервал (с 08:00 до 09:00), сформирует отчёты для обоих.

### 3. Отчёт по текущему нагрузочному тесту
```bash
./load_test_report.sh
```
Определит текущий тест, временные рамки, сформирует график нагрузки и детальные отчёты по сценариям.

### 4. Детальный отчёт по конкретному запросу
```bash
./queryid_report.sh 1234567890 "2026-03-31 10:00" "2026-03-31 11:00"
```
Создаст отдельные файлы для каждого типа ожидания с данными по истории выполнения запроса.

### 5. Чтение параметра из конфигурации (внутреннее использование)
```bash
./get_reports_param.sh /путь/к/reports.conf devices_list
```
Выведет значение `devices_list`, например `vdd vdc`.

## Примечания

- Все скрипты используют механизм `exit_code` для обработки ошибок и логирования в `.log` и `.err` файлы.
- Перед запуском убедитесь, что база данных `expecto_db` доступна, а пользователь `expecto_user` имеет права на выполнение необходимых функций.
- Отчёты сохраняются с правами `777`, чтобы быть доступными для других пользователей в системе.
- Для работы с iostat требуется, чтобы в конфигурации `reports.conf` был указан корректный список устройств (например, `devices_list = sda sdb`).
